## A. CSRF 與防偽

### 1. POST 動作沒有防偽驗證
❌ `[HttpPost] public ActionResult Transfer(...)` 沒有任何 token 檢查。 ✅ `[HttpPost, ValidateAntiForgeryToken]` 搭配表單中的 `@Html.AntiForgeryToken()`。
🎯 攻擊者的頁面自動送出一個跨站表單；受害者的驗證 cookie 會一併夾帶過去，於是轉帳就執行了。 📏 每一個會改變狀態的 POST 都要讓 `[ValidateAntiForgeryToken]` 與表單 token 成對出現。

### 2. token 出現在檢視中，但伺服器端沒有驗證
❌ 有放 `@Html.AntiForgeryToken()`，但動作缺少 `[ValidateAntiForgeryToken]`。 ✅ 兩半都要 —— 沒有屬性，這個欄位就是失效的。
🎯 隱藏的 `__RequestVerificationToken` 欄位有被輸出，卻從未被檢查，所以任何偽造的 POST 都會被接受。 📏 有渲染出 token 卻沒有伺服器端驗證，等於什麼都沒保護。

### 3. 會變更狀態的 GET 動作
❌ `public ActionResult Delete(int id) { db.Delete(id); }` 可透過 GET 觸及。 ✅ `[HttpPost, ValidateAntiForgeryToken]`；GET 維持唯讀。
🎯 電子郵件中的 `<img src="/user/delete/5">` 會在頁面載入時就觸發刪除 —— GET 的 CSRF 根本不需要表單。 📏 GET 絕不變更狀態；變更操作只走 POST 且要檢查 token。

### 4. AJAX POST 沒有帶防偽 token
❌ 從腳本發出 `$.post(url, data)` 卻沒有 token。 ✅ 從隱藏欄位／meta 讀取 token，並以 `__RequestVerificationToken`（表單欄位）或 `RequestVerificationToken` 標頭送出。
🎯 一個 JSON／AJAX 的變更端點完全跳過 CSRF，只因為沒人把 token 接進 XHR。 📏 AJAX 變更操作要像表單張貼一樣帶上防偽 token。

### 5. 驗證標頭 token，卻沒有在伺服器端讀取它
❌ 把 token 放在自訂標頭中送出，卻仍呼叫裸的 `[ValidateAntiForgeryToken]`，而它只會讀取表單欄位。 ✅ 一個自訂的 `ValidateHeaderAntiForgeryToken` 篩選器，呼叫 `AntiForgery.Validate(cookie, header)`。
🎯 預設屬性讀取 `Request.Form`，對 JSON 內容找不到任何東西，於是要嘛拋例外，要嘛（若設定錯誤）被略過。 📏 以標頭為基礎的 token 需要一個驗證標頭而非表單的篩選器。

### 6. `[ValidateAntiForgeryToken]` 在眾多變更之一上被遺漏
❌ 保護了 `Edit` 與 `Create`，卻漏掉較新的 `Archive` POST。 ✅ 稽核每一個 `[HttpPost]`；考慮採用全域篩選器，讓保護變成需明確退出。
🎯 那一個被忽略的變更操作就成了 CSRF 的入口。 📏 CSRF 審查要涵蓋*所有* POST 動作，而不只是顯而易見的那些。

### 7. 沒有全域防偽篩選器（靠記憶力）
❌ 在每個動作上手動加 `[ValidateAntiForgeryToken]`。 ✅ 註冊一個全域篩選器，要求所有不安全的動詞都帶 token，然後在真正需要之處以類似 `[AllowAnonymous]` 的方式明確退出。
🎯 一個新控制器上線時沒帶屬性，因為它是手動加的，而有人忘了。 📏 透過全域篩選器讓 CSRF 保護成為預設；要退出就明確退出。

### 8. 登入變更後防偽與 Identity 宣告不符
❌ 為匿名使用者產生的 token，在登入後未重新產生就被重複使用。 ✅ 把 `AntiForgeryConfig.UniqueClaimTypeIdentifier` 設為一個穩定的宣告（例如 Identity 的 user-id 宣告），讓 token 綁定到該使用者。
🎯 防偽 token 嵌入了使用者身分；若應用程式找不到你設定的宣告就會拋例外，或者一個來自不同身分的過期 token 會被接受。 📏 把 `UniqueClaimTypeIdentifier` 指向每個已驗證使用者都具備的宣告。

### 9. 因為驗證 cookie 名稱改變而使 token 被剝除
❌ 防偽 cookie 設定在整個伺服器陣列中與 OWIN 驗證 cookie 不一致。 ✅ 所有節點都採用一致的 `machineKey` 與 cookie 設定。
🎯 在沒有共用 `machineKey` 的網頁陣列上，節點 B 無法驗證節點 A 所核發的 token／cookie —— 要嘛讓正當使用者失敗，要嘛逼開發者弱化驗證。 📏 防偽需要在每一台伺服器上共用同一個 `machineKey`。

### 10. 為了「讓 AJAX 能動」而停用防偽
❌ 因為 SPA 呼叫回傳 500 就移除 `[ValidateAntiForgeryToken]`。 ✅ 去修正 token 的接線（`@Html.AntiForgeryToken()` → 標頭），而不是移除防護。
🎯 為了解開示範卡關而做的「暫時性」移除，變成了永久的 CSRF 漏洞。 📏 絕不為了修接線問題而刪掉 CSRF 防護。

## B. 模型繫結與過度張貼

### 11. 把請求直接繫結到 EF 實體上
❌ `public ActionResult Edit(User user) { db.Entry(user).State = Modified; }`。 ✅ 繫結到檢視模型／DTO，只對應表單所擁有的欄位。
🎯 使用者張貼 `IsAdmin=true`／`RoleId=1`；模型繫結器填滿實體，EF 就把這次權限提升持久化。 📏 絕不把 HTTP 輸入直接繫結到持久化實體上。

### 12. `TryUpdateModel`／`UpdateModel` 沒有 include 清單
❌ 對已取回的實體呼叫 `UpdateModel(user)`。 ✅ `TryUpdateModel(user, new[] { "Name", "Email" })` 或繫結到 DTO。
🎯 `UpdateModel` 會把*每一個*張貼進來的 key 繫結到實體上，因此多出來的表單欄位會覆寫受保護的欄位。 📏 把 `UpdateModel`／`TryUpdateModel` 限制在明確的屬性白名單內。

### 13. 把 `[Bind(Exclude=...)]` 當作防護
❌ 用 `[Bind(Exclude="IsAdmin")]` 來「阻擋」權限提升。 ✅ 以 `[Bind(Include=...)]` 白名單，或更好的做法是用一個專門打造的檢視模型。
🎯 黑名單會漏掉某人*下一個*加進來的敏感欄位（`RoleId`、`Balance`）；過度張貼又回來了。 📏 只白名單你接受的；絕不黑名單你害怕的。

### 14. `[Bind(Include=...)]` 放在型別上而非動作參數上
❌ 只依賴散落在模型類別上的 `[Bind]`。 ✅ 把 `[Bind(Include=...)]` 放在動作參數上以符合該動作的需求，或使用每個動作各自的 DTO。
🎯 共用模型上的一個 `[Bind]` 會全域套用，於是另一個需要不同欄位的動作，要嘛設不了那個欄位，要嘛重新打開過度張貼的缺口。 📏 把繫結的範圍限定在動作上，而非共用實體上。

### 15. 透過隱藏欄位對 `RoleId`／`IsActive` 進行批次指派
❌ 一個一般「編輯個人資料」表單的檢視模型卻包含了 `RoleId`。 ✅ 把自助服務 DTO（不含角色欄位）與管理員 DTO 分開。
🎯 自助服務表單從未顯示角色欄位，但繫結器接受了 POST 內容中一個精心構造的 `RoleId`。 📏 使用者不可設定的欄位，就不該存在於該動作所繫結的模型上。

### 16. 建立與管理員編輯共用同一個檢視模型
❌ 註冊表單與管理員畫面都用同一個 `UserVm`（含 `IsLockedOut`、`Roles`）。 ✅ 依權限層級使用不同的模型。
🎯 公開的註冊表單張貼了僅限管理員的欄位，而共用模型欣然接受繫結。 📏 每個授權情境對應一個繫結模型。

### 17. 繫結集合／巢狀物件卻沒有上限
❌ 從 `lines[0]..lines[N]` 繫結 `List<OrderLine>` 而沒有設上限。 ✅ 驗證數量與每個項目；限制集合大小。
🎯 攻擊者張貼 `lines[999999].X` 強迫巨量配置，或注入一個額外的巢狀實體圖，EF 接著就存了下去。 📏 被繫結的集合要有明確的最大值與逐項驗證。

### 18. 用 `[AllowHtml]` 讓欄位得以繫結，並當它已被清理
❌ `[AllowHtml] public string Bio;` 然後原樣渲染。 ✅ `[AllowHtml]` 只能搭配儲存／渲染前的伺服器端清理（AntiXSS／HtmlSanitizer）使用。
🎯 `[AllowHtml]` 只不過是對該屬性停用*請求驗證* —— 它不做任何清理，所以 `<script>` 會被繫結並在稍後渲染。 📏 `[AllowHtml]` 停用了一道防護；它從不增加防護 —— 你要自己清理。

### 19. 在請求所屬文化下繫結 `DateTime`／`decimal`
❌ 假設繫結器會照你預期的方式解析 `"1,5"` 或 `"03/04/2026"`。 ✅ 明確的格式／不變文化解析；驗證範圍。
🎯 另一個地區設定的使用者（或一個精心構造的 `Accept-Language`）會讓繫結器把日期／金額解析成與伺服器假設不同的結果。 📏 別讓被繫結的數值／日期解析取決於環境文化。

### 20. 模型繫結器默默忽略繫結失敗
❌ 假設一個 null／零的被繫結屬性代表「使用者留白」。 ✅ 檢查 `ModelState.IsValid` 以及每個 key 的 `ModelState` 錯誤。
🎯 `int Age` 上的型別不符會讓 `Age = 0` 並加上一個你從未檢查的 ModelState 錯誤，於是 `0` 就被當作資料存了下去。 📏 只有在 `ModelState.IsValid` 時，被繫結的值才可信賴。

## C. XSS 與 Razor

### 21. `@Html.Raw(userInput)`
❌ `@Html.Raw(Model.Comment)`。 ✅ 直接用 `@Model.Comment`（Razor 預設會做 HTML 編碼），若必須輸出 HTML 則先清理。
🎯 被儲存的 `<script>` 會在每個檢視者的已驗證工作階段中執行。 📏 對任何受使用者影響的字串使用 `@Html.Raw`，在證明已清理之前都算 XSS。

### 22. 以為 Razor 在 `<script>` 區塊內會做編碼
❌ `<script>var name = '@Model.Name';</script>`。 ✅ `@Html.Raw(Json.Encode(Model.Name))` 或一個 JSON 序列化的模型 —— 對 JS 字串情境而言，HTML 編碼是錯的。
🎯 `Model.Name = "'; stealCookies(); //"` 會跳脫出 JS 字串；Razor 的 HTML 編碼器擋不住 `'` 或 `</script>`。 📏 `<script>` 內的資料需要 JavaScript／JSON 編碼，不是 HTML 編碼。

### 23. HTML 屬性中未經編碼的值
❌ `<div data-x=@Model.Val>`（未加引號）。 ✅ 一律加引號：`<div data-x="@Model.Val">`。
🎯 未加引號的屬性讓 `Val = "x onmouseover=alert(1)"` 得以注入一個新屬性／事件處理常式。 📏 屬性值一律加引號，好讓編碼器能保護它們。

### 24. `@Html.Raw(Json.Encode(...))` 用在 HTML 情境中
❌ 把 `@Html.Raw(Json.Encode(model))` 渲染進 HTML 內文文字。 ✅ 只在 `<script>` 的值內使用它；在 HTML 中就讓 Razor 編碼。
🎯 `Json.Encode` 不會對 `</script>` 或 `<` 做 HTML 編碼，於是資料中的 `</script><script>` 酬載會關閉區塊並注入。 📏 讓編碼器對應到接收點：腳本用 JSON，標記用 HTML。

### 25. 從使用者資料建立 `MvcHtmlString`
❌ 在 helper 中 `return new HtmlString(userText);`。 ✅ 回傳已編碼的內容；只把可信賴、由程式產生的標記標記為安全。
🎯 一個把輸入包進 `MvcHtmlString`／`HtmlString` 的 HTML helper 等於告訴 Razor「別編碼這個」，把 XSS 送了出去。 📏 `MvcHtmlString` 是一個「內容已安全」的承諾 —— 只對你自己建構的內容做這件事。

### 26. `Url.Action`／`Url.Content` 的結果被丟進 JavaScript
❌ `<script>go('@Url.Action("Get", new { q = Request["q"] })')</script>`。 ✅ 對 URL 做 JSON 編碼，或在 JS 中用已編碼的元件組出它。
🎯 從輸入回顯的路由值會跳脫出 JS 字串情境。 📏 進入腳本的 URL 要做 JSON 編碼，不是原樣貼上。

### 27. 對 Markdown／富文字欄位使用 `@Html.Raw`
❌ 用 `@Html.Raw` 未經清理地渲染由使用者 Markdown 在伺服器端轉出的 HTML。 ✅ 在渲染前清理產生的 HTML（以允許清單限制標籤／屬性）。
🎯 Markdown 允許原始 HTML／`javascript:` 連結，它們會在轉換後存活並執行。 📏 轉換後的富文字要在抵達 `@Html.Raw` 之前先清理。

### 28. 把 `ViewBag`／`ViewData` 字串當作安全標記
❌ `@Html.Raw(ViewBag.Message)`，而 `Message` 來自查詢字串。 ✅ 預設編碼；把 `ViewBag` 的值當作不可信。
🎯 `?message=<script>...` 會流經 `ViewBag` 進入原始接收點，造成反射型 XSS。 📏 `ViewBag`／`ViewData` 的內容在接收點都是不可信的。

### 29. 為了接受富輸入而在全應用程式停用請求驗證
❌ `<httpRuntime requestValidationMode="2.0" />`，或在控制器上用 `[ValidateInput(false)]` 來接受 HTML。 ✅ 保持驗證開啟；把 `[AllowHtml]` 的範圍限定到那一個屬性並清理它。
🎯 在各處關掉請求驗證，會移除每個欄位那道最後防線的 `<` 過濾，擴大反射／儲存型 XSS。 📏 逐屬性放寬請求驗證，絕不全域放寬。

### 30. 編碼一次，但稍後又串接進標記
❌ `@Html.Encode(a) + "<b>" + Html.Encode(b)` 組成字串後再 `Raw` 渲染。 ✅ 讓 Razor 就地編碼每個內插；別用字串重新拼組 HTML。
🎯 兩段已編碼片段之間的空隙（你加上的那個字面 `<b>`），在整體稍後被 `Raw` 時就成了注入接縫。 📏 在接收點、在樣板中編碼 —— 而不是靠字串手術。

## D. 授權與篩選器

### 31. 缺少 `[Authorize]`（MVC 預設允許）
❌ 一個沒有 `[Authorize]` 的控制器動作。 ✅ 註冊一個全域 `AuthorizeAttribute` 篩選器，並對公開動作加 `[AllowAnonymous]`。
🎯 MVC 動作**預設是公開的**；一個被遺忘的屬性就把管理員動作送給了匿名使用者。 📏 預設拒絕：全域 `[Authorize]`、明確以 `[AllowAnonymous]` 退出。

### 32. 依賴隱藏的選單項目當作存取控制
❌ 在版面配置中隱藏「Admin」連結，而動作維持開放。 ✅ 在動作上強制 `[Authorize(Roles="Admin")]`。
🎯 URL 可被猜到／加入書籤；隱藏連結攔不住任何直接輸入它的人。 📏 隱藏 UI 是使用者體驗；動作屬性才是關卡。

### 33. 控制器層級的 `[AllowAnonymous]` 洩漏到新動作
❌ 在整個 `AccountController` 上加 `[AllowAnonymous]`。 ✅ 只加在 `Login`／`Register` 上，控制器維持 `[Authorize]`。
🎯 之後加進該控制器的動作會默默繼承匿名存取。 📏 逐動作、盡可能狹窄地授予匿名。

### 34. `[Authorize]` 在動作上，但子動作未受保護
❌ 一個 `[ChildActionOnly]` 的局部檢視渲染敏感資料卻沒有授權檢查。 ✅ 在子動作內或呼叫端控制器重新檢查授權。
🎯 `Html.Action("SecretWidget")` 渲染了特權資料，因為子動作本身從未被授權。 📏 子動作要強制自己的授權，而非父動作的。

### 35. 用 `User.IsInRole` 做角色檢查卻把角色名稱拼錯
❌ `if (User.IsInRole("Adminstrator"))`。 ✅ 從常數參照角色名稱；測試否定路徑。
🎯 拼錯的角色名稱永遠為 false（此處的 fail-open 意味著該區塊被略過，或關卡從不觸發）。 📏 角色名稱是常數，不是行內字串字面值。

### 36. `[Authorize(Users="...")]`／硬編碼身分
❌ `[Authorize(Users="alice,bob")]`。 ✅ 以資料存放區為後盾、以角色或原則為基礎的檢查。
🎯 這份清單會腐爛 —— 離職使用者仍保有存取權，新管理員被擋在外面，而修正之道是重新部署。 📏 依資料存放區中的角色／宣告授權，而非烘進程式碼的使用者名稱。

### 37. 在已授權、逐使用者的動作上用 `[OutputCache]`
❌ 在顯示當前使用者資料的頁面上用 `[Authorize][OutputCache(Duration=60)]`。 ✅ 別快取逐使用者的內容，或以身分做 `VaryByCustom` 並搭配 `Location=None`。
🎯 在時間窗內，使用者 A 的快取頁面被送給了使用者 B —— 跨使用者的資料洩漏。 📏 絕不在未依使用者變化的情況下輸出快取使用者專屬的已授權內容。

### 38. 授權篩選器用 `return` 而非設定 `Result`
❌ 自訂 `AuthorizeAttribute.OnAuthorization` 在失敗時只 `return;`。 ✅ 設定 `filterContext.Result = new HttpUnauthorizedResult()`。
🎯 沒設定結果就回傳，會讓管線繼續下去，動作就在未授權下執行。 📏 失敗的授權篩選器必須設定一個短路的 `Result`，不能只是回傳。

### 39. `AuthorizeAttribute.AuthorizeCore` 被覆寫成 fail open
❌ 在自訂授權檢查中 `catch { return true; }`。 ✅ Fail closed —— 任何錯誤都拒絕。
🎯 角色查詢期間資料存放區打了個嗝，結果是授予存取而非拒絕。 📏 授權錯誤要拒絕；絕不在例外時預設為 true。

### 40. Web API 控制器用錯 `Authorize` 型別
❌ 在 `ApiController` 上用 `System.Web.Mvc.AuthorizeAttribute`（或反過來）。 ✅ Web API 用 `System.Web.Http.AuthorizeAttribute`，MVC 用 `System.Web.Mvc`。
🎯 MVC 篩選器在 Web API 管線中根本不會執行，於是該 API 端點實際上未受保護。 📏 讓 `Authorize` 的命名空間對應到管線（MVC 對 Web API）。

## E. Identity 與 OWIN

### 41. 把 Identity 的密碼雜湊器換成快速雜湊
❌ 把 `IPasswordHasher` 換成裸的 `SHA256`。 ✅ 保留預設的 `PasswordHasher`（PBKDF2）—— 參見 [範例 01](./01-password-hashing-unsalted-sha256.zh-TW.md)。
🎯 一張外洩的 `AspNetUsers` 表若採用快速、未加鹽的雜湊，幾分鐘內就被破解。 📏 別降級 Identity 內建的 PBKDF2 雜湊器。

### 42. 登入路徑上未啟用鎖定
❌ `SignInManager.PasswordSignInAsync(user, pw, isPersistent, shouldLockout: false)`。 ✅ 傳入 `shouldLockout: true` 並設定 `MaxFailedAccessAttemptsBeforeLockout`。
🎯 鎖定關閉時，憑證填充／暴力破解會對每個帳號無節流地執行。 📏 啟用 Identity 鎖定，並在登入時遵循它。

### 43. 管理員帳號沒有 2FA 選項
❌ 管理員只靠密碼保護。 ✅ 為特權角色透過 `UserManager` 的 token 提供者啟用 `TwoFactorAuthentication`。
🎯 一個被釣魚／被重複使用的管理員密碼就是完全淪陷，沒有第二因素可以阻止。 📏 高權限帳號需要第二因素。

### 44. 每次登入都 `isPersistent: true`
❌ 總是核發一個持久的「記住我」cookie。 ✅ 只在使用者要求時才持久化；其餘維持工作階段 cookie。
🎯 共用／公用機器上的持久 cookie 在瀏覽器關閉後仍存活，被下一個人重複使用。 📏 持久驗證 cookie 要是選擇加入，絕非預設。

### 45. 生產環境用 `CookieSecure = SameAsRequest`
❌ 在 OWIN cookie 上保留 `CookieSecureOption.SameAsRequest`。 ✅ 在 HTTPS 之後用 `CookieSecure = CookieSecureOption.Always`。
🎯 一個 HTTP 請求（或一個被剝除的連結）就會以明文送出驗證 cookie，給在途攻擊者可乘之機。 📏 生產環境的驗證 cookie 要 `Secure = Always`。

### 46. `SecurityStampValidator` 間隔太長（無法撤銷）
❌ `OnValidateIdentity(validateInterval: TimeSpan.FromDays(30))` 或乾脆省略它。 ✅ 一個短間隔（例如數分鐘），好讓戳記變更撤銷工作階段。
🎯 密碼變更／停用在間隔過去之前並不會使攻擊者的 cookie 失效。 📏 短的 `SecurityStampValidator` 間隔，好讓憑證變更切斷仍在運作的工作階段。

### 47. 密碼變更沒有輪替安全戳記
❌ 直接更新雜湊，繞過 `UserManager.UpdateSecurityStampAsync`。 ✅ 走 `ChangePasswordAsync`，它會推進戳記。
🎯 受害者在遭入侵後重設，但被竊的 cookie 仍能運作，因為戳記從未改變。 📏 憑證變更必須更新安全戳記。

### 48. 開放的外部登入流程沒有 state／`returnUrl` 檢查
❌ 信任來自外部回呼的 `returnUrl` 並重新導向到它。 ✅ 重新導向前先 `Url.IsLocalUrl(returnUrl)`；驗證登入提供者的 state。
🎯 OAuth 回呼重新導向到攻擊者的 URL，讓釣魚或 token 擷取得逞。 📏 驗證外部登入的回傳 URL 與提供者 state。

### 49. `ExpireTimeSpan` 巨大且開著 `SlidingExpiration`
❌ `ExpireTimeSpan = 90 天` + `SlidingExpiration = true`。 ✅ 一個短窗口；滑動會續期它，所以被竊的 cookie 不應能用上好幾個月。
🎯 外洩的 cookie 在整個長窗口內維持有效，而滑動會在它被使用時持續續期。 📏 讓 cookie 生命週期短；別讓滑動到期造出近乎不死的工作階段。

### 50. 登入／註冊可透過 HTTP 觸及
❌ `AccountController` 上沒有 `[RequireHttps]`；OWIN 允許 HTTP。 ✅ `[RequireHttps]`（或全域 HTTPS 重新導向）加上 `Secure` cookie。
🎯 憑證與剛核發的驗證 cookie 在首次請求時以明文傳輸。 📏 驗證端點僅限 HTTPS。

## F. 動作與動作結果

### 51. 透過 GET 提供的 `JsonResult` 洩漏資料
❌ `return Json(sensitive, JsonRequestBehavior.AllowGet);`。 ✅ 透過 POST 回傳 JSON，或根本別以 GET 暴露敏感資料。
🎯 `AllowGet` 讓跨站的 `<script src>`／JSON 劫持得以讀取酬載；它也會落入快取／記錄中。 📏 別對承載敏感或逐使用者資料的 JSON 使用 `AllowGet`。

### 52. 透過 `Redirect(returnUrl)` 造成開放式重新導向
❌ 直接從查詢參數 `return Redirect(returnUrl);`。 ✅ `if (Url.IsLocalUrl(returnUrl)) return Redirect(returnUrl); return RedirectToAction("Index");`。
🎯 `?returnUrl=https://evil.tld` 讓你自己的登入頁面成為以假亂真的釣魚跳板。 📏 只重新導向到 `Url.IsLocalUrl` 認可的 URL。

### 53. `[HttpGet]` 動作執行狀態變更
❌ `[HttpGet] public ActionResult Approve(int id)`。 ✅ `[HttpPost, ValidateAntiForgeryToken]`。
🎯 一個會預取的瀏覽器或一個 `<img>` 標籤就在使用者毫無意圖下觸發了核准。 📏 狀態變更走 POST；GET 要安全／冪等。

### 54. 從使用者提供的路徑建構 `File()`
❌ `return File(Server.MapPath("~/files/" + name), ...)`。 ✅ 先解析，再驗證正規化後的路徑仍在預期根目錄之下；使用 id→路徑對應。
🎯 `name = "..\\..\\web.config"` 會透過路徑遍歷讀取機密。 📏 絕不從原始輸入建構所提供的檔案路徑；限制在一個根目錄內。

### 55. `File(path, contentType, fileName)` 帶未清理的下載名稱
❌ 把使用者文字直接當作下載檔名傳入。 ✅ 清理檔名；剝除 CR/LF 與路徑分隔符。
🎯 檔名中的 CR/LF 會注入標頭（回應分割）或偽造副檔名。 📏 清理 `Content-Disposition` 檔名。

### 56. `Content()` 從輸入回傳 HTML
❌ `return Content("<div>" + userText + "</div>", "text/html");`。 ✅ 做編碼，或回傳一個由 Razor 編碼的檢視／模型。
🎯 `ContentResult` 不做任何編碼，於是輸入變成反射型 XSS。 📏 帶 `text/html` 的 `Content()` 是原始接收點 —— 在它之前先編碼。

### 57. `Redirect` 到原始的 `Request` 值
❌ `Redirect(Request.QueryString["url"])`。 ✅ 以 `Url.IsLocalUrl` 或主機的允許清單驗證。
🎯 與 #52 相同的開放式重新導向濫用，只是直接取自請求。 📏 每個重新導向目標都要驗證，無論它來自何處。

### 58. `Server.TransferRequest`／`Server.Execute` 帶輸入路徑
❌ 轉移到一個由使用者輸入組成的路徑。 ✅ 路由到已知動作；絕不轉移到由輸入衍生的路徑。
🎯 攻擊者將伺服器端執行導向一個非預期的處理常式，繞過路由層級的授權。 📏 伺服器端轉移只以固定、已授權的端點為目標。

### 59. 從 JSON 動作回傳實體物件
❌ `return Json(dbUser)` 序列化整個 EF 實體。 ✅ 投影到一個只含用戶端所需欄位的 DTO。
🎯 延遲載入／被序列化的導覽屬性以及像 `PasswordHash` 這樣的欄位會一併搭進回應。 📏 JSON 回應序列化 DTO，絕不序列化原始實體。

### 60. 把跨重新導向的 `TempData` 當作防竄改狀態信任
❌ 儲存一個授權決策（`TempData["CanEdit"]=true`）並在下一個請求信任它。 ✅ 在目標動作重新推導／重新檢查。
🎯 `TempData` 是便利狀態，不是安全邊界 —— 決策必須在伺服器端重新驗證。 📏 每個請求都重新檢查授權；別把它帶在 `TempData` 裡。

## G. 驗證與錯誤處理

### 61. 沒有檢查 `ModelState.IsValid`
❌ 使用被繫結的值卻沒有 `if (!ModelState.IsValid) return View(model);`。 ✅ 每個 POST 都以 `ModelState.IsValid` 把關。
🎯 無效／惡意的輸入越過了 `[Required]`／`[Range]`，因為那個旗標從未被檢視。 📏 在 `ModelState.IsValid` 通過之前，不進行任何寫入。

### 62. 只有用戶端驗證（unobtrusive JS）
❌ 依賴 jQuery Validate／data 屬性來強制執行。 ✅ 同一套 `DataAnnotations` 要透過 `ModelState` 在伺服器端強制執行。
🎯 一個直接的 POST（curl／Fiddler）完全跳過瀏覽器與被停用的／JS 檢查。 📏 用戶端驗證是使用者體驗；伺服器要重新驗證每個欄位。

### 63. 用 `[ValidateInput(false)]` 來接受特殊字元
❌ 在動作上關掉請求驗證以允許 `<` 或 `&`。 ✅ 保持開啟；在那一個屬性上用 `[AllowHtml]` 並清理它。
🎯 對整個動作停用內建的 XSS 前置過濾，會讓每個欄位都對腳本注入敞開。 📏 逐屬性放寬請求驗證，絕不整個動作一次性放寬。

### 64. 生產環境用 `<customErrors mode="Off">`
❌ 留著 `mode="Off"` 讓死亡黃頁顯示出來。 ✅ `mode="RemoteOnly"`（或 `On`）搭配一個友善的錯誤檢視。
🎯 未處理的例外會把完整的堆疊追蹤、檔案路徑與 SQL 渲染給匿名使用者 —— 免費的偵查情報。 📏 生產環境開啟自訂錯誤；細節只進記錄。

### 65. 沒有全域例外處理／`HandleErrorAttribute` 的缺口
❌ 假設每個動作都有 try/catch。 ✅ 一個全域的 `HandleErrorAttribute`／`Application_Error`，記錄並顯示一個通用檢視。
🎯 未受保護動作中的一個未處理例外會洩漏內部資訊或不優雅地 500。 📏 要有一個全域錯誤處理器，記錄細節並回傳一個通用頁面。

### 66. `HandleErrorAttribute` 只在自訂錯誤開啟時才有效
❌ 期望在 `customErrors` 為 `Off` 時 `[HandleError]` 仍能顯示 `Error.cshtml`。 ✅ 啟用自訂錯誤；`[HandleError]` 會遵循該設定。
🎯 開發者以為有 `[HandleError]` 罩著，但在 `customErrors="Off"` 下原始例外仍會顯示。 📏 `[HandleError]` 取決於 `<customErrors>` 有被啟用。

### 67. 把訊息回顯給用戶端的例外篩選器
❌ 在錯誤回應中回傳 `ex.Message`／`ex.ToString()`。 ✅ 給使用者通用訊息；把完整細節給記錄。
🎯 `ex.Message` 中的 SQL 文字、連線字串與路徑成了攻擊者的地圖。 📏 用戶端拿通用錯誤；記錄拿細節。

### 68. `Application_Error` 吞掉例外卻沒有 `Server.ClearError`／記錄
❌ 空的 `Application_Error`，或一個沒記錄就重新導向的。 ✅ 帶上下文記錄例外，然後 `Server.ClearError()` 並顯示錯誤頁面。
🎯 失敗默默消失 —— 包括失敗的安全檢查 —— 在遭入侵後也無從調查。 📏 `Application_Error` 一律在清除／重新導向之前先記錄。

### 69. ELMAH 端點暴露且未受保護
❌ `elmah.axd` 可遠端觸及且沒有驗證。 ✅ 限制它（`<location>` + `[Authorize]`／IP 允許清單）；設定 `<security allowRemoteAccess="false">`。
🎯 ELMAH 的儀表板會把完整例外 —— 查詢字串、cookie、含憑證的表單張貼 —— 重播給任何找到該 URL 的人。 📏 鎖定 ELMAH；它儲存敏感的請求傾印。

### 70. 洩漏存在性／內部資訊的驗證訊息
❌ 「該電子郵件已被註冊」／「該電子郵件沒有帳號」。 ✅ 對帳號存在性流程給予通用、一致的訊息。
🎯 註冊與重設表單成了使用者列舉的預言機。 📏 帳號存在性的回應要一致且不洩漏。

## H. web.config 與部署

### 71. 生產環境用 `<compilation debug="true">`
❌ 帶著 `debug="true"` 出貨。 ✅ 發行部署用 `debug="false"`。
🎯 除錯模式會停用逾時、膨脹記憶體、輸出冗長錯誤，並略過批次最佳化 —— 是 DoS 與資訊洩漏的放大器。 📏 每一份部署的 web.config 都要 `debug="false"`。

### 72. 共用／弱／被提交的 `machineKey`
❌ 從部落格複製並提交的 `machineKey`，或跨陣列自動產生。 ✅ 每個應用程式一把強而唯一的 key，存為受保護的機密，且在陣列節點間一致。
🎯 已知的 `machineKey` 讓攻擊者能偽造驗證 cookie 與防偽 token，並竄改 ViewState／受保護資料。 📏 `machineKey` 是每個應用程式一把的強機密，只在陣列內共用。

### 73. web.config 中明文的連線字串
❌ `<connectionStrings>` 中明文的 SQL 憑證。 ✅ 用 `aspnet_regiis -pe` 加密該區段，或使用整合式安全性。
🎯 任何對已部署設定（或儲存庫）有讀取權的人都能拿到資料庫密碼。 📏 加密設定機密，或使用 Windows 整合式驗證。

### 74. 全應用程式的 `requestValidationMode="2.0"`
❌ 用 `<httpRuntime requestValidationMode="2.0" />` 來閃避驗證錯誤。 ✅ 保持 4.x 模式；用 `[AllowHtml]` 界定例外的範圍。
🎯 2.0 模式會延遲／限制請求驗證，各處都弱化了內建的反射型 XSS 前置過濾。 📏 保持 4.x 請求驗證；別全域退回 2.0。

### 75. 缺少安全標頭
❌ 沒有 `X-Content-Type-Options`、`X-Frame-Options`、`Content-Security-Policy`。 ✅ 透過 `<customHeaders>` 或回應篩選器加上它們。
🎯 透過框架的點擊劫持、把上傳內容 MIME 探測成腳本，以及不受限的 XSS 酬載。 📏 出貨一套安全回應標頭的基準線。

### 76. `<httpCookies>` 沒有 `httpOnlyCookies`／`requireSSL`
❌ 預設的 cookie 設定。 ✅ `<httpCookies httpOnlyCookies="true" requireSSL="true" />`。
🎯 XSS 從 `document.cookie` 讀取工作階段／驗證 cookie，或者它在 HTTP 請求中洩漏。 📏 透過設定讓 cookie 為 HttpOnly 且僅限 SSL。

### 77. 因錯誤的 `defaultRedirect` 而洩漏的自訂錯誤
❌ `<customErrors mode="On" defaultRedirect="~/error?msg=...">` 回顯例外。 ✅ 靜態的通用錯誤頁面；在伺服器端記錄細節。
🎯 那個「友善」的錯誤頁面透過其查詢字串反映了攻擊者可控或內部的細節。 📏 錯誤頁面不顯示任何衍生自例外或輸入的東西。

### 78. 目錄瀏覽／提供靜態機密
❌ 讓 `.bak`、`web.config.old` 或 `App_Data` 檔案可經網頁觸及。 ✅ 以 `<location>`／處理常式對應封鎖；讓機密遠離可服務的資料夾。
🎯 攻擊者下載 `web.config.bak` 並讀取連線字串與 `machineKey`。 📏 沒有任何敏感內容位於可經網頁服務的路徑之下。

### 79. 生產環境關閉 `EnableOptimizations` 的組合包
❌ 把 `BundleTable.EnableOptimizations = false` 出貨到生產環境。 ✅ 為發行啟用最佳化；驗證組合包有壓縮／加上版本。
🎯 除了膨脹，未加版本的個別檔案還會破壞快取破除，並可能提供陳舊／重複的腳本。 📏 生產環境開啟組合包最佳化。

### 80. CDN 組合包後備但無完整性驗證
❌ `bundles.UseCdn = true` 指向第三方 CDN，沒有後備／SRI。 ✅ 提供本機後備，並在 CDN 支援時優先使用 SRI。
🎯 一個遭入侵或被劫持的 CDN 會把惡意腳本送進你來源下的每一頁。 📏 CDN 託管的腳本需要後備與完整性驗證。

## I. EF6 與資料層

### 81. 以字串串接建構 `Database.SqlQuery`
❌ `db.Database.SqlQuery<T>("SELECT * FROM U WHERE Name='" + name + "'")`。 ✅ 參數化：`SqlQuery<T>("... WHERE Name=@p0", name)`。
🎯 `name = "' OR 1=1--"` 會傾印整張表 —— 原始 SQL 繞過了 EF 的參數化。 📏 原始 EF SQL 使用 `@p` 參數，絕不串接。

### 82. `ExecuteSqlCommand` 帶內插輸入
❌ `db.Database.ExecuteSqlCommand($"DELETE FROM Log WHERE Id={id}")`。 ✅ `ExecuteSqlCommand("DELETE FROM Log WHERE Id=@p0", id)`（或 `SqlParameter`）。
🎯 內插的命令會像串接的查詢一樣被注入。 📏 每一個 `ExecuteSqlCommand` 都要參數化。

### 83. `DbContext` 跨請求共用／作為單例
❌ 一個靜態／單例的 `DbContext`。 ✅ 每個請求／工作單元一個 `DbContext`，在結束時釋放。
🎯 `DbContext` 不是執行緒安全的；並行請求會破壞它的變更追蹤器並在使用者之間滲漏資料。 📏 `DbContext` 逐請求且會被釋放。

### 84. 延遲載入在檢視中造成 N+1
❌ 在開著延遲載入的情況下於 Razor 中迭代 `order.Lines`。 ✅ `.Include(o => o.Lines)`（積極載入）或在查詢中投影到 DTO。
🎯 檢視渲染時每一列都觸發一次新查詢 —— 在真實資料下是數百次往返（而且若上下文已被釋放就會失敗）。 📏 事先載入檢視所需的內容；別在 Razor 中延遲載入。

### 85. 被更新的實體沒有並行 token
❌ 對取回的實體盲目 `SaveChanges`。 ✅ 一個 `[Timestamp] byte[] RowVersion`，讓 EF 把它加進 `WHERE` 並拋出 `DbUpdateConcurrencyException`。
🎯 兩位管理員編輯同一筆記錄；第二位默默覆寫了第一位（遺失更新）。 📏 會被並行編輯的實體要帶一個 `[Timestamp]` 並行 token。

### 86. 把實體直接回傳給檢視，洩漏欄位
❌ `return View(dbUser)` 以實體作為模型。 ✅ 一個只暴露顯示欄位的檢視模型。
🎯 `PasswordHash`／內部旗標就在模型物件裡，可能在渲染輸出、JSON 或除錯工具中浮現。 📏 檢視繫結到檢視模型，而非原始實體。

### 87. 對你接著要修改並儲存的實體用 `AsNoTracking()`
❌ `.AsNoTracking()` 後又編輯並呼叫 `SaveChanges`。 ✅ 追蹤你打算更新的實體；`AsNoTracking` 只用於唯讀查詢。
🎯 對一個不追蹤實體的變更不會被偵測到，於是 `SaveChanges` 默默什麼都沒持久化 —— 一個資料遺失的臭蟲。 📏 `AsNoTracking` 是唯讀；寫入用被追蹤的實體。

### 88. 多步驟寫入沒有交易
❌ 兩次 `SaveChanges` 呼叫，沒有 `TransactionScope`／`Database.BeginTransaction`。 ✅ 把相關的寫入包在一個交易裡。
🎯 兩次儲存之間的一次當機留下寫了一半、不一致的資料。 📏 相關的寫入要原子性地認可。

### 89. 信任資料庫唯一／FK 違規永不浮現
❌ 沒有處理包裝 SQL 2601/2627/547 的 `DbUpdateException`。 ✅ 攔截它，對應到一個乾淨的 409／驗證錯誤。
🎯 一個重複的 key 變成透過錯誤頁面洩漏 SQL 細節的原始 500。 📏 把已知的資料庫條件約束錯誤轉譯成面向使用者的結果。

### 90. 無上限的查詢餵給檢視／格線
❌ `db.Orders.ToList()` 回傳每一列。 ✅ `.Skip().Take()` 分頁 + 有上限的頁面大小。
🎯 一張長大的表回傳數百萬列，把工作處理程序 OOM，並讓頁面卡死。 📏 每個清單查詢都要分頁且有上限。

## J. .NET Framework 執行階段陷阱

### 91. `async void` 的動作或 helper
❌ `public async void Save(...)`。 ✅ `public async Task<ActionResult> Save(...)` —— MVC 5 支援回傳 `Task` 的非同步動作。
🎯 來自 `async void` 的例外無法被管線攔截，會使工作處理程序當機；而且該動作無法被 await。 📏 MVC 動作用 `async Task`，絕不用 `async void`。

### 92. 在動作中對非同步用 `.Result`／`.Wait()`（死結）
❌ 在控制器內 `var x = svc.GetAsync().Result;`。 ✅ `var x = await svc.GetAsync();` 並讓該動作變成非同步。
🎯 傳統 ASP.NET 有一個 `SynchronizationContext`；在負載下對非同步阻塞會使請求執行緒死結。 📏 一路 `await` 上去；絕不在 MVC 中對非同步阻塞。

### 93. 函式庫／服務層缺少 `ConfigureAwait(false)`
❌ 函式庫的 await 不必要地擷取了 ASP.NET 上下文。 ✅ 在非 UI 的函式庫程式碼中用 `ConfigureAwait(false)`。
🎯 在每個 await 上擷取請求上下文，會在傳統 `SynchronizationContext` 下餵養同一種死結／飢餓。 📏 函式庫的 await 使用 `ConfigureAwait(false)`。

### 94. 每個請求都 `new HttpClient()`
❌ 在每個動作中建構一個 `HttpClient`（傳統 MVC 沒有 `IHttpClientFactory`）。 ✅ 一個單一的 `static`／共用 `HttpClient`，在應用程式生命週期內重複使用。
🎯 每個實例都佔著一個處於 `TIME_WAIT` 的 socket；在負載下這台機器會耗盡連接埠，呼叫開始失敗。 📏 重複使用一個長生命週期的 `HttpClient`。

### 95. 用 `DateTime.Now` 做儲存／到期邏輯
❌ 持久化／比較 `DateTime.Now`。 ✅ `DateTime.UtcNow`；只在顯示時才轉成當地時間。
🎯 日光節約時間／時區／伺服器搬遷的偏移會破壞 token 到期、排程與稽核排序。 📏 以 UTC 儲存與比較。

### 96. 對輸入或 key 做文化敏感的解析／格式化
❌ 在請求文化下 `decimal.Parse(s)`／`DateTime.Parse(s)`／`ToUpper()`。 ✅ `CultureInfo.InvariantCulture`；邏輯用 `StringComparison.Ordinal`。
🎯 逗號當小數點的地區設定會把金額解析錯；土耳其語的 I 會破壞一個本意為序數比較的比較。 📏 機器邏輯用不變／序數；文化只用於顯示。

### 97. 用 `Random`／`Guid.NewGuid()` 做安全 token
❌ 用 `new Random().Next()` 或一個 GUID 當作密碼重設／電子郵件確認 token。 ✅ `RNGCryptoServiceProvider.GetBytes(32)`（或 Identity 的 token 提供者）。
🎯 `Random` 從幾個輸出就可預測，GUID 也不保證不可猜測，所以 token 可被偽造。 📏 安全 token 來自 CSPRNG，而非 `Random`／`Guid`。

### 98. 控制器／服務中硬編碼的機密
❌ 程式碼中 `const string ApiKey = "sk_live_...";`。 ✅ `<appSettings>`（加密）或一個機密存放區，在執行階段讀取。
🎯 任何有儲存庫或歷史存取權的人 —— `git log -p` —— 就永遠擁有這把 key。 📏 機密絕不存在於原始碼或 git 歷史中。

### 99. 記錄 PII／token／完整請求傾印
❌ `log.Info("body=" + Request.Form.ToString())` 或記錄驗證 cookie。 ✅ 記錄 id 與已遮蔽的標記；絕不整份記錄內容。
🎯 密碼、token 與 PII 堆積在記錄檔與錯誤追蹤器中，並隨之一起外洩。 📏 絕不記錄 token、密碼或完整的請求內容。

### 100. 控制器／服務中可變的 `static` 共用狀態
❌ 一個逐請求被變動、沒有同步的 `static Dictionary` 快取。 ✅ 逐請求範圍，或一個執行緒安全／不可變的結構（`ConcurrentDictionary`）。
🎯 並行請求爭用該共用集合，將它破壞或把一個使用者的資料洩漏給另一個。 📏 網頁應用程式中不要有未同步的可變靜態成員。
