# 04 — ASP.NET MVC 5 (.NET Framework): 100 cautionary samples

A fast-reference catalogue of **100 pitfalls specific to ASP.NET MVC 5 on the classic .NET Framework
(4.x)** — Razor views, ASP.NET Identity, and OWIN cookie auth. Same teaching spirit as
[01](./01-password-hashing-unsalted-sha256.md): each shows the dangerous shape, the scenario in which it
bites, and the rule. These are **MVC-5 idioms** — not ASP.NET Core, not Web Forms.

Compressed on purpose. When one bites for real, promote it to a full before/after file like sample 01.

**Legend:** ❌ = don't · ✅ = do · 🎯 = the scenario that makes it dangerous · 📏 = the rule.

## A. CSRF & anti-forgery

### 1. POST action with no anti-forgery validation
❌ `[HttpPost] public ActionResult Transfer(...)` with no token check. ✅ `[HttpPost, ValidateAntiForgeryToken]` + `@Html.AntiForgeryToken()` in the form.
🎯 An attacker's page auto-submits a cross-site form; the victim's auth cookie rides along and the transfer runs. 📏 Every state-changing POST pairs `[ValidateAntiForgeryToken]` with a form token.

### 2. Token in the view but not validated on the server
❌ `@Html.AntiForgeryToken()` present, but the action lacks `[ValidateAntiForgeryToken]`. ✅ Both halves — the field is inert without the attribute.
🎯 The hidden `__RequestVerificationToken` field is emitted but never checked, so any forged POST is accepted. 📏 A rendered token with no server-side validation protects nothing.

### 3. GET action that mutates state
❌ `public ActionResult Delete(int id) { db.Delete(id); }` reachable by GET. ✅ `[HttpPost, ValidateAntiForgeryToken]`; GET stays read-only.
🎯 `<img src="/user/delete/5">` in an email fires the delete on page load — CSRF needs no form for a GET. 📏 GET never changes state; mutations are POST-only and token-checked.

### 4. AJAX POST without the anti-forgery token
❌ `$.post(url, data)` from a script with no token. ✅ Read the token from a hidden field/meta and send it as `__RequestVerificationToken` (form field) or the `RequestVerificationToken` header.
🎯 A JSON/AJAX mutation endpoint skips CSRF entirely because nobody wired the token into the XHR. 📏 AJAX mutations carry the anti-forgery token just like form posts.

### 5. Validating the header token without reading it server-side
❌ Send the token in a custom header but still call bare `[ValidateAntiForgeryToken]`, which only reads the form field. ✅ A custom `ValidateHeaderAntiForgeryToken` filter that calls `AntiForgery.Validate(cookie, header)`.
🎯 The default attribute reads `Request.Form`, finds nothing for a JSON body, and either throws or (if misconfigured) is skipped. 📏 Header-based tokens need a filter that validates the header, not the form.

### 6. `[ValidateAntiForgeryToken]` forgotten on one of many mutations
❌ Protect `Edit` and `Create` but not the newer `Archive` POST. ✅ Audit every `[HttpPost]`; consider a global filter so protection is opt-out.
🎯 The one overlooked mutation becomes the CSRF entry point. 📏 CSRF review covers *all* POST actions, not the obvious ones.

### 7. No global anti-forgery filter (relying on memory)
❌ Add `[ValidateAntiForgeryToken]` by hand on each action. ✅ Register a global filter that requires the token for all unsafe verbs, then `[AllowAnonymous]`-style opt out where truly needed.
🎯 A new controller ships without the attribute because it was manual and someone forgot. 📏 Make CSRF protection the default via a global filter; opt out explicitly.

### 8. Anti-forgery + Identity claim mismatch after login change
❌ Tokens generated for an anonymous user reused after sign-in without regeneration. ✅ Set `AntiForgeryConfig.UniqueClaimTypeIdentifier` to a stable claim (e.g. Identity's user-id claim) so tokens bind to the user.
🎯 The anti-forgery token embeds a user identity; if the app can't find your configured claim it throws, or a stale token from a different identity is accepted. 📏 Point `UniqueClaimTypeIdentifier` at a claim every authenticated user has.

### 9. Token stripped because the auth cookie name changed
❌ Anti-forgery cookie config diverging from the OWIN auth cookie across a farm. ✅ Consistent `machineKey` and cookie config across all nodes.
🎯 On a web farm without a shared `machineKey`, node B can't validate the token/cookie node A issued — either failing legit users or forcing devs to weaken validation. 📏 Anti-forgery needs a shared `machineKey` across every server.

### 10. Disabling anti-forgery "to make AJAX work"
❌ Removing `[ValidateAntiForgeryToken]` because the SPA call 500s. ✅ Fix the token plumbing (`@Html.AntiForgeryToken()` → header) instead of removing the guard.
🎯 The "temporary" removal to unblock a demo becomes the permanent CSRF hole. 📏 Never delete the CSRF guard to fix a wiring problem.

## B. Model binding & over-posting

### 11. Binding the request straight onto an EF entity
❌ `public ActionResult Edit(User user) { db.Entry(user).State = Modified; }`. ✅ Bind a view model/DTO, map only the fields the form owns.
🎯 The user posts `IsAdmin=true`/`RoleId=1`; the model binder fills the entity and EF persists the escalation. 📏 Never bind HTTP input directly onto a persistence entity.

### 12. `TryUpdateModel`/`UpdateModel` with no include list
❌ `UpdateModel(user)` over a fetched entity. ✅ `TryUpdateModel(user, new[] { "Name", "Email" })` or bind a DTO.
🎯 `UpdateModel` binds *every* posted key onto the entity, so extra form fields overwrite protected columns. 📏 Constrain `UpdateModel`/`TryUpdateModel` to an explicit property whitelist.

### 13. Trusting `[Bind(Exclude=...)]` as the guard
❌ `[Bind(Exclude="IsAdmin")]` to "block" escalation. ✅ Whitelist with `[Bind(Include=...)]` or, better, a purpose-built view model.
🎯 A blacklist misses the *next* sensitive field someone adds (`RoleId`, `Balance`); over-posting returns. 📏 Whitelist what you accept; never blacklist what you fear.

### 14. `[Bind(Include=...)]` on the type instead of the action parameter
❌ Relying on `[Bind]` scattered on the model class only. ✅ Put `[Bind(Include=...)]` on the action parameter for that action's needs, or use per-action DTOs.
🎯 One `[Bind]` on the shared model applies globally, so a different action that needs another field either can't set it or reopens over-posting. 📏 Scope binding to the action, not a shared entity.

### 15. Mass assignment of `RoleId`/`IsActive` via hidden fields
❌ A view model that includes `RoleId` for an ordinary "edit profile" form. ✅ Separate self-service DTO (no role fields) from the admin DTO.
🎯 The self-service form never showed a role box, but the binder accepts a crafted `RoleId` in the POST body. 📏 A field the user may not set must not exist on that action's bound model.

### 16. Reusing one view model for create and admin edit
❌ Same `UserVm` (with `IsLockedOut`, `Roles`) for both the signup form and the admin screen. ✅ Distinct models per privilege level.
🎯 The public signup posts admin-only fields the shared model happily binds. 📏 One bound model per authorization context.

### 17. Binding collections/nested objects without bounds
❌ `List<OrderLine>` bound from `lines[0]..lines[N]` with no cap. ✅ Validate count and each item; cap the collection size.
🎯 An attacker posts `lines[999999].X` and forces huge allocation, or injects an extra nested entity graph EF then saves. 📏 Bound collections have an explicit maximum and per-item validation.

### 18. `[AllowHtml]` to let a field bind, treating it as sanitized
❌ `[AllowHtml] public string Bio;` and rendering it raw. ✅ `[AllowHtml]` only with server-side sanitization (AntiXSS/HtmlSanitizer) before store/render.
🎯 `[AllowHtml]` merely disables *request validation* for that property — it does no cleaning, so `<script>` binds and later renders. 📏 `[AllowHtml]` disables a guard; it never adds one — sanitize yourself.

### 19. Binding `DateTime`/`decimal` under the request's culture
❌ Assuming the binder parses `"1,5"` or `"03/04/2026"` the way you expect. ✅ Explicit formats / invariant parsing; validate ranges.
🎯 A user in another locale (or a crafted `Accept-Language`) makes the binder parse a date/amount differently than the server assumes. 📏 Don't let bound numeric/date parsing depend on ambient culture.

### 20. Model binder silently ignoring bind failures
❌ Assuming a null/zero bound property means "user left it blank". ✅ Check `ModelState.IsValid` and per-key `ModelState` errors.
🎯 A type-mismatch on `int Age` leaves `Age = 0` and adds a ModelState error you never checked, so `0` is saved as data. 📏 A bound value is only trustworthy when `ModelState.IsValid`.

## C. XSS & Razor

### 21. `@Html.Raw(userInput)`
❌ `@Html.Raw(Model.Comment)`. ✅ Plain `@Model.Comment` (Razor HTML-encodes by default), or sanitize first if HTML is required.
🎯 Stored `<script>` runs in every viewer's authenticated session. 📏 `@Html.Raw` on any user-influenced string is XSS until proven sanitized.

### 22. Assuming Razor encodes inside a `<script>` block
❌ `<script>var name = '@Model.Name';</script>`. ✅ `@Html.Raw(Json.Encode(Model.Name))` or a JSON-serialized model — HTML-encoding is wrong for a JS string context.
🎯 `Model.Name = "'; stealCookies(); //"` breaks out of the JS string; Razor's HTML encoder doesn't stop `'` or `</script>`. 📏 Data inside `<script>` needs JavaScript/JSON encoding, not HTML encoding.

### 23. Unencoded value in an HTML attribute
❌ `<div data-x=@Model.Val>` (unquoted). ✅ Always quote: `<div data-x="@Model.Val">`.
🎯 An unquoted attribute lets `Val = "x onmouseover=alert(1)"` inject a new attribute/event handler. 📏 Attribute values are always quoted so the encoder can protect them.

### 24. `@Html.Raw(Json.Encode(...))` used in an HTML context
❌ Rendering `@Html.Raw(Json.Encode(model))` into HTML body text. ✅ Use it only inside a `<script>` value; in HTML, let Razor encode.
🎯 `Json.Encode` doesn't HTML-encode `</script>` or `<`, so a `</script><script>` payload in the data closes the block and injects. 📏 Match the encoder to the sink: JSON for script, HTML for markup.

### 25. Building a `MvcHtmlString` from user data
❌ `return new HtmlString(userText);` in a helper. ✅ Return encoded content; only mark trusted, generated markup as safe.
🎯 An HTML helper that wraps input in `MvcHtmlString`/`HtmlString` tells Razor "don't encode this" and ships the XSS. 📏 `MvcHtmlString` is a promise the content is already safe — only make it for content you built.

### 26. `Url.Action`/`Url.Content` result dropped into JavaScript
❌ `<script>go('@Url.Action("Get", new { q = Request["q"] })')</script>`. ✅ JSON-encode the URL, or build it in JS from encoded components.
🎯 Route values echoed from input break out of the JS string context. 📏 URLs headed into script are JSON-encoded, not pasted raw.

### 27. `@Html.Raw` of a Markdown/rich-text field
❌ Rendering server-converted HTML from user Markdown with `@Html.Raw`, unsanitized. ✅ Sanitize the generated HTML (allow-list tags/attributes) before rendering.
🎯 Markdown allows raw HTML/`javascript:` links that survive conversion and execute. 📏 Converted rich text is sanitized before it reaches `@Html.Raw`.

### 28. Trusting `ViewBag`/`ViewData` strings as safe markup
❌ `@Html.Raw(ViewBag.Message)` where `Message` came from a query string. ✅ Encode by default; treat `ViewBag` values as untrusted.
🎯 `?message=<script>...` flows through `ViewBag` into a raw sink for reflected XSS. 📏 `ViewBag`/`ViewData` content is untrusted at the sink.

### 29. Disabling request validation app-wide to allow rich input
❌ `<httpRuntime requestValidationMode="2.0" />` or `[ValidateInput(false)]` on a controller to accept HTML. ✅ Keep validation on; scope `[AllowHtml]` to the one property and sanitize it.
🎯 Turning off request validation everywhere removes the last-ditch `<` filter for every field, widening reflected/stored XSS. 📏 Relax request validation per-property, never globally.

### 30. Encoding once but concatenating into markup later
❌ `@Html.Encode(a) + "<b>" + Html.Encode(b)` assembled as a string then `Raw`-rendered. ✅ Let Razor encode each interpolation in place; don't rebuild HTML from strings.
🎯 A gap between two encoded pieces (the literal `<b>` you added) becomes the injection seam when the whole thing is later `Raw`ed. 📏 Encode at the sink, in the template — not by string surgery.

## D. Authorization & filters

### 31. Missing `[Authorize]` (MVC default-allow)
❌ A controller action with no `[Authorize]`. ✅ Register a global `AuthorizeAttribute` filter and `[AllowAnonymous]` the public actions.
🎯 MVC actions are **public by default**; a forgotten attribute ships an admin action to anonymous users. 📏 Default-deny: global `[Authorize]`, explicit `[AllowAnonymous]` opt-out.

### 32. Relying on a hidden menu item as the access control
❌ Hiding the "Admin" link in the layout and leaving the action open. ✅ Enforce `[Authorize(Roles="Admin")]` on the action.
🎯 The URL is guessable/bookmarkable; hiding the link stops nobody who types it. 📏 Hiding UI is UX; the action attribute is the gate.

### 33. `[AllowAnonymous]` at controller scope leaking to new actions
❌ `[AllowAnonymous]` on the whole `AccountController`. ✅ Put it only on `Login`/`Register`, keep the controller `[Authorize]`.
🎯 A later action added to that controller silently inherits anonymous access. 📏 Grant anonymity per-action, as narrowly as possible.

### 34. `[Authorize]` on the action but child actions unguarded
❌ A `[ChildActionOnly]` partial that renders sensitive data with no auth check. ✅ Re-check authorization inside the child action or the calling controller.
🎯 `Html.Action("SecretWidget")` renders privileged data because the child action itself was never authorized. 📏 Child actions enforce their own authorization, not the parent's.

### 35. Role check via `User.IsInRole` with a typo'd role name
❌ `if (User.IsInRole("Adminstrator"))`. ✅ Reference role names from constants; test the negative path.
🎯 A misspelled role name is always false (fail-open here means the block is skipped, or a gate never triggers). 📏 Role names are constants, not inline string literals.

### 36. `[Authorize(Users="...")]` / hardcoded identities
❌ `[Authorize(Users="alice,bob")]`. ✅ Role- or policy-based checks backed by the store.
🎯 The list rots — a departed user keeps access, a new admin is locked out and the fix is a redeploy. 📏 Authorize by role/claim from the store, not baked-in usernames.

### 37. `[OutputCache]` on an authorized, per-user action
❌ `[Authorize][OutputCache(Duration=60)]` on a page showing the current user's data. ✅ Don't cache per-user content, or `VaryByCustom` on identity with `Location=None`.
🎯 User A's cached page is served to user B within the window — a cross-user data leak. 📏 Never output-cache user-specific authorized content without varying by user.

### 38. Authorization filter that returns instead of setting `Result`
❌ Custom `AuthorizeAttribute.OnAuthorization` that just `return;`s on failure. ✅ Set `filterContext.Result = new HttpUnauthorizedResult()`.
🎯 Returning without setting a result lets the pipeline continue and the action runs unauthorized. 📏 A failed auth filter must set a short-circuiting `Result`, not just return.

### 39. `AuthorizeAttribute.AuthorizeCore` overridden to fail open
❌ `catch { return true; }` in a custom authorize check. ✅ Fail closed — any error denies.
🎯 A store hiccup during the role lookup grants access instead of denying it. 📏 Authorization errors deny; never default-true on exception.

### 40. Web API controllers using the wrong `Authorize` type
❌ `System.Web.Mvc.AuthorizeAttribute` on an `ApiController` (or vice-versa). ✅ Use `System.Web.Http.AuthorizeAttribute` for Web API, `System.Web.Mvc` for MVC.
🎯 The MVC filter never runs in the Web API pipeline, so the API endpoint is effectively unprotected. 📏 Match the `Authorize` namespace to the pipeline (MVC vs Web API).

## E. Identity & OWIN

### 41. Replacing the Identity password hasher with a fast hash
❌ Swapping `IPasswordHasher` for a bare `SHA256`. ✅ Keep the default `PasswordHasher` (PBKDF2) — see [sample 01](./01-password-hashing-unsalted-sha256.md).
🎯 A leaked `AspNetUsers` table with fast, unsalted hashes cracks in minutes. 📏 Don't downgrade Identity's built-in PBKDF2 hasher.

### 42. Lockout not enabled on the sign-in path
❌ `SignInManager.PasswordSignInAsync(user, pw, isPersistent, shouldLockout: false)`. ✅ Pass `shouldLockout: true` and configure `MaxFailedAccessAttemptsBeforeLockout`.
🎯 With lockout off, credential stuffing/brute force runs unthrottled against every account. 📏 Enable Identity lockout and honor it on sign-in.

### 43. No 2FA option for admin accounts
❌ Admins protected by password only. ✅ Enable `TwoFactorAuthentication` via `UserManager` token providers for privileged roles.
🎯 A phished/reused admin password is a full compromise with no second factor to stop it. 📏 High-privilege accounts require a second factor.

### 44. `isPersistent: true` for every login
❌ Always issuing a persistent "remember me" cookie. ✅ Persist only when the user asked; keep session cookies for the rest.
🎯 A persistent cookie on a shared/kiosk machine survives the browser closing and is reused by the next person. 📏 Persistent auth cookies are opt-in, never the default.

### 45. `CookieSecure = SameAsRequest` in production
❌ Leaving `CookieSecureOption.SameAsRequest` on the OWIN cookie. ✅ `CookieSecure = CookieSecureOption.Always` behind HTTPS.
🎯 A single HTTP request (or a stripped link) sends the auth cookie in cleartext for an on-path attacker. 📏 Auth cookies are `Secure = Always` in production.

### 46. `SecurityStampValidator` interval too long (no revocation)
❌ `OnValidateIdentity(validateInterval: TimeSpan.FromDays(30))` or omitting it. ✅ A short interval (e.g. minutes) so stamp changes revoke sessions.
🎯 A password change / disable doesn't invalidate the attacker's cookie until the interval elapses. 📏 Short `SecurityStampValidator` interval so credential changes cut live sessions.

### 47. Password change doesn't rotate the security stamp
❌ Updating the hash directly, bypassing `UserManager.UpdateSecurityStampAsync`. ✅ Go through `ChangePasswordAsync`, which bumps the stamp.
🎯 The victim resets after a breach but the stolen cookie keeps working because the stamp never changed. 📏 Credential changes must update the security stamp.

### 48. Open external-login flow without state/`returnUrl` checks
❌ Trusting `returnUrl` from the external callback and redirecting to it. ✅ `Url.IsLocalUrl(returnUrl)` before redirecting; validate the login provider state.
🎯 The OAuth callback redirects to an attacker URL, enabling phishing or token capture. 📏 Validate external-login return URLs and provider state.

### 49. `ExpireTimeSpan` huge with `SlidingExpiration` on
❌ `ExpireTimeSpan = 90 days` + `SlidingExpiration = true`. ✅ A short window; sliding renews it, so a stolen cookie shouldn't be valid for months.
🎯 A leaked cookie stays valid for the whole long window and sliding keeps renewing it as it's used. 📏 Keep the cookie lifetime short; don't let sliding expiration create near-immortal sessions.

### 50. Login/registration reachable over HTTP
❌ No `[RequireHttps]` on `AccountController`; OWIN allowing HTTP. ✅ `[RequireHttps]` (or global HTTPS redirect) plus `Secure` cookies.
🎯 Credentials and the freshly issued auth cookie travel in cleartext on first request. 📏 Auth endpoints are HTTPS-only.

## F. Actions & action results

### 51. `JsonResult` served over GET leaking data
❌ `return Json(sensitive, JsonRequestBehavior.AllowGet);`. ✅ Return JSON via POST, or don't expose sensitive data by GET at all.
🎯 `AllowGet` lets a cross-site `<script src>`/JSON hijack read the payload; it also lands in caches/logs. 📏 Don't `AllowGet` on JSON that carries sensitive or per-user data.

### 52. Open redirect via `Redirect(returnUrl)`
❌ `return Redirect(returnUrl);` straight from a query param. ✅ `if (Url.IsLocalUrl(returnUrl)) return Redirect(returnUrl); return RedirectToAction("Index");`.
🎯 `?returnUrl=https://evil.tld` powers convincing phishing off your own login page. 📏 Only redirect to URLs `Url.IsLocalUrl` approves.

### 53. `[HttpGet]` action performing a state change
❌ `[HttpGet] public ActionResult Approve(int id)`. ✅ `[HttpPost, ValidateAntiForgeryToken]`.
🎯 A prefetching browser or an `<img>` tag triggers the approval with no user intent. 📏 State changes are POST; GET is safe/idempotent.

### 54. `File()` built from a user-supplied path
❌ `return File(Server.MapPath("~/files/" + name), ...)`. ✅ Resolve, then verify the canonical path stays under the intended root; use an id→path map.
🎯 `name = "..\\..\\web.config"` reads secrets via path traversal. 📏 Never build a served file path from raw input; confine to a root.

### 55. `File(path, contentType, fileName)` with an unsanitized download name
❌ Passing user text straight as the download filename. ✅ Sanitize the filename; strip CR/LF and path separators.
🎯 CR/LF in the filename injects headers (response splitting) or spoofs the extension. 📏 Sanitize `Content-Disposition` filenames.

### 56. `Content()` returning HTML from input
❌ `return Content("<div>" + userText + "</div>", "text/html");`. ✅ Encode, or return a view/model that Razor encodes.
🎯 `ContentResult` does no encoding, so input becomes reflected XSS. 📏 `Content()` with `text/html` is a raw sink — encode before it.

### 57. `Redirect` to a raw `Request` value
❌ `Redirect(Request.QueryString["url"])`. ✅ Validate against `Url.IsLocalUrl` or an allow-list of hosts.
🎯 Same open-redirect abuse as #52, just sourced directly off the request. 📏 Every redirect target is validated, wherever it came from.

### 58. `Server.TransferRequest`/`Server.Execute` with input path
❌ Transferring to a path assembled from user input. ✅ Route to known actions; never transfer to input-derived paths.
🎯 An attacker steers server-side execution to an unintended handler, bypassing route-level auth. 📏 Server-side transfers target fixed, authorized endpoints only.

### 59. Returning entity objects from a JSON action
❌ `return Json(dbUser)` serializing the full EF entity. ✅ Project to a DTO with only the fields the client needs.
🎯 Lazy-loaded/serialized navigation properties and columns like `PasswordHash` ride along into the response. 📏 JSON responses serialize DTOs, never raw entities.

### 60. Trusting `TempData` across a redirect as tamper-proof state
❌ Storing an authorization decision (`TempData["CanEdit"]=true`) and trusting it on the next request. ✅ Re-derive/re-check on the target action.
🎯 `TempData` is convenience state, not a security boundary — decisions must be re-verified server-side. 📏 Re-check authorization on each request; don't carry it in `TempData`.

## G. Validation & error handling

### 61. Not checking `ModelState.IsValid`
❌ Using bound values without `if (!ModelState.IsValid) return View(model);`. ✅ Guard every POST on `ModelState.IsValid`.
🎯 Invalid/malicious input flows past `[Required]`/`[Range]` because the flag was never inspected. 📏 No write proceeds until `ModelState.IsValid` passes.

### 62. Client-side validation only (unobtrusive JS)
❌ Relying on jQuery Validate / data-attributes for enforcement. ✅ The same `DataAnnotations` are enforced server-side via `ModelState`.
🎯 A direct POST (curl/Fiddler) skips the browser and the disabled/JS checks entirely. 📏 Client validation is UX; the server re-validates every field.

### 63. `[ValidateInput(false)]` to accept special characters
❌ Turning off request validation on an action to allow `<` or `&`. ✅ Keep it on; use `[AllowHtml]` on the one property + sanitize.
🎯 Disabling the built-in XSS pre-filter for the whole action opens every field to script injection. 📏 Relax request validation per-property, never per-action wholesale.

### 64. `<customErrors mode="Off">` in production
❌ Leaving `mode="Off"` so the yellow screen of death shows. ✅ `mode="RemoteOnly"` (or `On`) with a friendly error view.
🎯 Unhandled exceptions render full stack traces, file paths, and SQL to anonymous users — free recon. 📏 Custom errors on in production; details go to logs only.

### 65. No global exception handling / `HandleErrorAttribute` gap
❌ Assuming every action try/catches. ✅ A global `HandleErrorAttribute`/`Application_Error` that logs and shows a generic view.
🎯 An unhandled exception in an unguarded action leaks internals or 500s ungracefully. 📏 Have one global error handler that logs detail and returns a generic page.

### 66. `HandleErrorAttribute` only works with custom errors on
❌ Expecting `[HandleError]` to show `Error.cshtml` while `customErrors` is `Off`. ✅ Enable custom errors; `[HandleError]` honors that setting.
🎯 Devs think they're covered by `[HandleError]`, but with `customErrors="Off"` the raw exception still shows. 📏 `[HandleError]` depends on `<customErrors>` being enabled.

### 67. Exception filter that echoes the message to the client
❌ Returning `ex.Message`/`ex.ToString()` in the error response. ✅ Generic message to the user; full detail to the log.
🎯 SQL text, connection strings, and paths in `ex.Message` become an attacker's map. 📏 Clients get generic errors; logs get the detail.

### 68. `Application_Error` swallowing without `Server.ClearError`/logging
❌ Empty `Application_Error` or one that redirects without logging. ✅ Log the exception with context, then `Server.ClearError()` and show the error page.
🎯 Failures vanish silently — including failed security checks — and can't be investigated after a breach. 📏 `Application_Error` always logs before it clears/redirects.

### 69. ELMAH endpoint exposed and unsecured
❌ `elmah.axd` reachable remotely with no auth. ✅ Restrict it (`<location>` + `[Authorize]`/IP allow-list); set `<security allowRemoteAccess="false">`.
🎯 ELMAH's dashboard replays full exceptions — query strings, cookies, form posts with credentials — to anyone who finds the URL. 📏 Lock down ELMAH; it stores sensitive request dumps.

### 70. Validation messages that reveal existence/internals
❌ "That email is already registered" / "No account for that email". ✅ Generic, uniform messaging for account-existence flows.
🎯 Registration and reset forms become a user-enumeration oracle. 📏 Account-existence responses are uniform and non-revealing.

## H. web.config & deployment

### 71. `<compilation debug="true">` in production
❌ Shipping with `debug="true"`. ✅ `debug="false"` for release deployments.
🎯 Debug mode disables timeouts, bloats memory, emits verbose errors, and skips batch optimization — a DoS and info-leak amplifier. 📏 `debug="false"` in every deployed web.config.

### 72. Shared/weak/committed `machineKey`
❌ A `machineKey` copied from a blog and committed, or auto-gen across a farm. ✅ A strong, unique key per app, stored as a protected secret, consistent across farm nodes.
🎯 A known `machineKey` lets an attacker forge auth cookies and anti-forgery tokens and tamper with ViewState/protected data. 📏 `machineKey` is a strong per-app secret, shared only within the farm.

### 73. Connection strings in cleartext in web.config
❌ Plaintext SQL credentials in `<connectionStrings>`. ✅ `aspnet_regiis -pe` to encrypt the section, or use integrated security.
🎯 Anyone with read access to the deployed config (or the repo) gets the DB password. 📏 Encrypt config secrets or use Windows integrated auth.

### 74. `requestValidationMode="2.0"` app-wide
❌ `<httpRuntime requestValidationMode="2.0" />` to dodge validation errors. ✅ Keep the 4.x mode; scope exceptions with `[AllowHtml]`.
🎯 The 2.0 mode delays/limits request validation, weakening the built-in reflected-XSS pre-filter everywhere. 📏 Keep 4.x request validation; don't globally revert to 2.0.

### 75. Missing security headers
❌ No `X-Content-Type-Options`, `X-Frame-Options`, `Content-Security-Policy`. ✅ Add them via `<customHeaders>` or a response filter.
🎯 Clickjacking via framing, MIME-sniffing an upload into script, and unconstrained XSS payloads. 📏 Ship a baseline of security response headers.

### 76. `<httpCookies>` without `httpOnlyCookies`/`requireSSL`
❌ Default cookie config. ✅ `<httpCookies httpOnlyCookies="true" requireSSL="true" />`.
🎯 XSS reads the session/auth cookie from `document.cookie`, or it leaks over an HTTP request. 📏 Cookies are HttpOnly and SSL-only by config.

### 77. Custom errors that leak via a wrong `defaultRedirect`
❌ `<customErrors mode="On" defaultRedirect="~/error?msg=...">` echoing the exception. ✅ Static generic error page; log details server-side.
🎯 The "friendly" error page reflects attacker-controlled or internal detail through its query string. 📏 The error page shows nothing derived from the exception or input.

### 78. Directory browsing / static secrets served
❌ Leaving `.bak`, `web.config.old`, or `App_Data` files web-reachable. ✅ Block by `<location>`/handler mappings; keep secrets out of servable folders.
🎯 An attacker downloads `web.config.bak` and reads connection strings and the `machineKey`. 📏 Nothing sensitive lives under a web-servable path.

### 79. Bundling with `EnableOptimizations` off in production
❌ `BundleTable.EnableOptimizations = false` shipped to prod. ✅ Enable optimization for release; verify bundles minify/version.
🎯 Beyond bloat, un-versioned individual files defeat cache-busting and can serve stale/duplicated scripts. 📏 Bundling optimization is on in production.

### 80. CDN bundle fallback without integrity
❌ `bundles.UseCdn = true` pointing at a third-party CDN, no fallback/SRI. ✅ Provide a local fallback and prefer SRI where the CDN supports it.
🎯 A compromised or hijacked CDN serves malicious script into every page under your origin. 📏 CDN-hosted scripts need a fallback and integrity verification.

## I. EF6 & data layer

### 81. `Database.SqlQuery` built by string concatenation
❌ `db.Database.SqlQuery<T>("SELECT * FROM U WHERE Name='" + name + "'")`. ✅ Parameterize: `SqlQuery<T>("... WHERE Name=@p0", name)`.
🎯 `name = "' OR 1=1--"` dumps the table — raw SQL bypasses EF's parameterization. 📏 Raw EF SQL uses `@p` parameters, never concatenation.

### 82. `ExecuteSqlCommand` with interpolated input
❌ `db.Database.ExecuteSqlCommand($"DELETE FROM Log WHERE Id={id}")`. ✅ `ExecuteSqlCommand("DELETE FROM Log WHERE Id=@p0", id)` (or `SqlParameter`s).
🎯 Interpolated commands inject just like concatenated queries. 📏 Parameterize every `ExecuteSqlCommand`.

### 83. `DbContext` shared across requests / as a singleton
❌ A static/singleton `DbContext`. ✅ One `DbContext` per request/unit of work, disposed at the end.
🎯 `DbContext` isn't thread-safe; concurrent requests corrupt its change tracker and bleed data between users. 📏 `DbContext` is per-request and disposed.

### 84. Lazy loading causing N+1 in the view
❌ Iterating `order.Lines` in Razor with lazy loading on. ✅ `.Include(o => o.Lines)` (eager) or project to a DTO in the query.
🎯 Each row triggers a fresh query while the view renders — hundreds of round-trips under real data (and it fails if the context is already disposed). 📏 Load what the view needs up front; don't lazy-load in Razor.

### 85. No concurrency token on updated entities
❌ Blind `SaveChanges` on a fetched entity. ✅ A `[Timestamp] byte[] RowVersion` so EF adds it to the `WHERE` and throws `DbUpdateConcurrencyException`.
🎯 Two admins edit the same record; the second silently overwrites the first (lost update). 📏 Concurrently edited entities carry a `[Timestamp]` concurrency token.

### 86. Returning entities straight to the view, leaking columns
❌ `return View(dbUser)` with the entity as the model. ✅ A view model exposing only display fields.
🎯 `PasswordHash`/internal flags are in the model object and can surface in rendered output, JSON, or debugging tools. 📏 Views bind to view models, not raw entities.

### 87. `AsNoTracking()` on entities you then modify and save
❌ `.AsNoTracking()` then editing and calling `SaveChanges`. ✅ Track entities you intend to update; use `AsNoTracking` for read-only queries only.
🎯 Changes to a no-tracking entity aren't detected, so `SaveChanges` silently persists nothing — a data-loss bug. 📏 `AsNoTracking` is read-only; tracked entities for writes.

### 88. Multi-step writes without a transaction
❌ Two `SaveChanges` calls with no `TransactionScope`/`Database.BeginTransaction`. ✅ Wrap related writes in one transaction.
🎯 A crash between the two saves leaves half-written, inconsistent data. 📏 Related writes commit atomically.

### 89. Trusting DB unique/FK violations to never surface
❌ No handling for `DbUpdateException` wrapping SQL 2601/2627/547. ✅ Catch it, map to a clean 409/validation error.
🎯 A duplicate key becomes a raw 500 that leaks SQL detail via the error page. 📏 Translate known DB constraint errors into user-facing results.

### 90. Unbounded queries feeding a view/grid
❌ `db.Orders.ToList()` returning every row. ✅ `.Skip().Take()` paging + a capped page size.
🎯 A grown table returns millions of rows, OOMs the worker process, and hangs the page. 📏 Every list query is paged with a maximum.

## J. .NET Framework runtime pitfalls

### 91. `async void` action or helper
❌ `public async void Save(...)`. ✅ `public async Task<ActionResult> Save(...)` — MVC 5 supports async actions returning `Task`.
🎯 Exceptions from `async void` can't be caught by the pipeline and crash the worker process; the action can't be awaited. 📏 `async Task`, never `async void`, in MVC actions.

### 92. `.Result`/`.Wait()` on async in an action (deadlock)
❌ `var x = svc.GetAsync().Result;` inside a controller. ✅ `var x = await svc.GetAsync();` and make the action async.
🎯 Classic ASP.NET has a `SynchronizationContext`; blocking on async deadlocks the request thread under load. 📏 `await` all the way up; never block on async in MVC.

### 93. Missing `ConfigureAwait(false)` in library/service layers
❌ Library awaits that capture the ASP.NET context needlessly. ✅ `ConfigureAwait(false)` in non-UI library code.
🎯 Capturing the request context on every await feeds the same deadlock/starvation under the legacy `SynchronizationContext`. 📏 Library awaits use `ConfigureAwait(false)`.

### 94. `new HttpClient()` per request
❌ Constructing an `HttpClient` in each action (no `IHttpClientFactory` in classic MVC). ✅ A single `static`/shared `HttpClient` reused for the app's lifetime.
🎯 Each instance holds a socket in `TIME_WAIT`; under load the box exhausts ports and calls start failing. 📏 Reuse one long-lived `HttpClient`.

### 95. `DateTime.Now` for storage/expiry logic
❌ Persisting/comparing `DateTime.Now`. ✅ `DateTime.UtcNow`; convert to local only for display.
🎯 DST/timezone/server-move drift breaks token expiry, scheduling, and audit ordering. 📏 Store and compare in UTC.

### 96. Culture-sensitive parse/format on input or keys
❌ `decimal.Parse(s)` / `DateTime.Parse(s)` / `ToUpper()` under the request culture. ✅ `CultureInfo.InvariantCulture`; `StringComparison.Ordinal` for logic.
🎯 A comma-decimal locale mis-parses money; Turkish-I breaks an ordinal-intended compare. 📏 Invariant/ordinal for machine logic; culture only for display.

### 97. `Random`/`Guid.NewGuid()` for security tokens
❌ `new Random().Next()` or a GUID as a password-reset/email-confirm token. ✅ `RNGCryptoServiceProvider.GetBytes(32)` (or Identity's token providers).
🎯 `Random` is predictable from a few outputs and GUIDs aren't guaranteed unguessable, so tokens can be forged. 📏 Security tokens come from a CSPRNG, not `Random`/`Guid`.

### 98. Hardcoded secrets in controllers/services
❌ `const string ApiKey = "sk_live_...";` in code. ✅ `<appSettings>` (encrypted) or a secret store, read at runtime.
🎯 Anyone with repo or history access — `git log -p` — owns the key forever. 📏 Secrets never live in source or git history.

### 99. Logging PII/tokens/full request dumps
❌ `log.Info("body=" + Request.Form.ToString())` or logging the auth cookie. ✅ Log ids and redacted markers; never bodies wholesale.
🎯 Passwords, tokens, and PII pile up in log files and error trackers and breach along with them. 📏 Never log tokens, passwords, or full request bodies.

### 100. Mutable `static` shared state in a controller/service
❌ A `static Dictionary` cache mutated per request without synchronization. ✅ Per-request scope, or a thread-safe/immutable structure (`ConcurrentDictionary`).
🎯 Concurrent requests race the shared collection, corrupting it or leaking one user's data to another. 📏 No unsynchronized mutable statics in a web app.
