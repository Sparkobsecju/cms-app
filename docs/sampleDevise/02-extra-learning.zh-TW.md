## A. 加密與機密資訊

### 1. 直接用快速雜湊處理密碼
❌ 用 `SHA256.HashData(pw)` / MD5 / SHA1 儲存密碼。✅ 使用 PBKDF2/bcrypt/Argon2 搭配每位使用者專屬的鹽值（salt）（見範例 01）。
🎯 資料庫外洩 → GPU 每秒可破解數十億次，未加鹽的雜湊還會被彩虹表（rainbow table）攻破。📏 密碼雜湊必須 **緩慢 + 加鹽**。

### 2. 用 `==` 比較機密/雜湊/權杖
❌ `if (userToken == storedToken)`。✅ `CryptographicOperations.FixedTimeEquals(a, b)`。
🎯 字串 `==` 在第一個相異位元組就短路；時間差會讓機密逐位元組洩漏。📏 所有機密資料都要用固定時間（constant-time）比較。

### 3. 用 `Random`/`Guid.NewGuid()` 產生安全權杖
❌ 用 `new Random().Next()` 或 GUID 當作密碼重設權杖。✅ `RandomNumberGenerator.GetBytes(32)`。
🎯 `Random` 只要看幾個輸出就能預測；GUID 也不保證無法被猜中。📏 任何不能被攻擊者預測的東西都要用 CSPRNG（密碼學安全的亂數產生器）。

### 4. 在原始碼中寫死金鑰/機密
❌ `var key = "s3cr3t-signing-key";`。✅ 執行期設定（本應用程式從資料庫的 `SysConfig` 讀取 JWT 金鑰）。
🎯 任何能讀取版本庫/歷史紀錄的人都能永久偽造權杖。📏 機密絕不放進程式碼或 git 歷史紀錄。

### 5. 沒有完整性保護的加密（未驗證模式）
❌ 使用沒有 MAC 的 AES-CBC。✅ AES-GCM（`AesGcm`）或先加密再加 MAC（encrypt-then-MAC）。
🎯 填充預言（padding-oracle）/位元翻轉（bit-flipping）攻擊會悄悄竄改密文。📏 使用具驗證的加密（AEAD）。

### 6. 重複使用 nonce/IV
❌ 在 GCM/CTR 中使用固定或重設計數器的 IV。✅ 每則訊息都用全新的隨機 nonce，並與密文一起儲存。
🎯 在 GCM 中重複使用 nonce 會洩漏驗證金鑰；在 CTR 中則會把兩段明文 XOR 在一起。📏 同一把金鑰下絕不重複使用 nonce。

### 7. 自己土法煉鋼寫加密
❌ 用 XOR/base64/位移自製「加密」。✅ 使用 `System.Security.Cryptography` 中經過驗證的原語。
🎯 自製方案在標準密碼分析下幾分鐘就被攻破。📏 別發明加密；使用函式庫。

### 8. 記錄機密
❌ `_logger.LogInformation("token={Token}", jwt)`。✅ 記錄一個識別碼或遮蔽後的標記。
🎯 機密會落到記錄檔/彙整系統/截圖裡，比金鑰輪替存活得更久。📏 絕不記錄權杖、密碼、金鑰或完整的個人識別資訊（PII）。

### 9. 把機密放在 URL／查詢字串
❌ `GET /reset?token=abc`。✅ 把權杖放在請求主體或短期有效的標頭中。
🎯 查詢字串會出現在存取記錄、瀏覽器歷史紀錄與 `Referer` 標頭中。📏 機密放在主體/標頭，絕不放在 URL。

### 10. 過短／低熵的簽章金鑰
❌ 8 個字元的 HMAC 金鑰。✅ ≥32 位元組（本應用程式強制 `MinKeyBytes = 32`，採失敗即拒絕（fail-closed））。
🎯 短金鑰可被暴力破解，讓攻擊者能簽出有效的權杖。📏 HMAC-SHA256 需要 ≥256 位元的金鑰。

### 11. 機密先被提交，之後才在後續提交中「移除」
❌ 在新的提交中刪掉金鑰就以為它消失了。✅ 輪替機密；清理歷史紀錄（BFG/filter-repo）。
🎯 Git 保留每個歷史版本；舊金鑰只要一個 `git log -p` 就能取得。📏 外洩的機密就等於報廢 — 立刻輪替。

### 12. 以不分大小寫／帶文化設定的方式比較密碼雜湊
❌ `hash.Equals(other, StringComparison.OrdinalIgnoreCase)`。✅ 位元組層級的序數（ordinal）／固定時間比較。
🎯 大小寫摺疊會擴大符合的範圍；文化規則會造成因地區設定而異的錯誤。📏 序數、位元組精確、固定時間。

## B. 身分驗證、工作階段與 JWT

### 13. 未驗證簽發者/受眾/有效期
❌ 接受任何簽章正確的 JWT。✅ 驗證 `iss`、`aud`、`exp` 與簽章金鑰（本應用程式四項全設）。
🎯 為另一個共用金鑰的服務所簽發的權杖也會被接受。📏 你所依賴的每一項聲明（claim）都要驗證。

### 14. 接受 `alg: none` 或演算法混淆
❌ 信任權杖的 `alg` 標頭。✅ 固定允許的演算法；只用對稱式可避免 RS256→HS256 混淆。
🎯 攻擊者設定 `alg:none`，或用公鑰當作機密以 HS256 簽章。📏 由伺服器決定演算法，而不是權杖。

### 15. 長效權杖且無法撤銷
❌ 效期 30 天的 JWT，無法作廢。✅ 短效的存取權杖 + 更新/輪替；密碼變更時更新安全戳記（security-stamp）。
🎯 外洩的權杖在整個效期內都有效；被攻陷的帳號無法被切斷。📏 短效期 + 一條撤銷途徑。

### 16. 變更密碼未使舊工作階段失效
❌ 變更密碼後，既有的 JWT 仍然有效。✅ 遞增每位使用者的「安全戳記」，並在驗證時檢查。
🎯 受害者在遭入侵後變更密碼，攻擊者偷來的權杖卻依然可用。📏 憑證變更必須終止使用中的工作階段。

### 17. 透過不同錯誤訊息列舉使用者
❌「查無此使用者」對比「密碼錯誤」；對未知使用者有不同的回應時間。✅ 所有失敗都回傳同一個通用的 401。
🎯 攻擊者摸清有效的使用者名稱，再鎖定它們進行密碼噴灑（spraying）。📏 驗證失敗必須無法區分。

### 18. 登入未設限流/鎖定
❌ 無限次的登入嘗試。✅ 依帳號與 IP 分別限流 + 退避/鎖定。
🎯 憑證填充（credential stuffing）/暴力破解可暢行無阻（搭配快速雜湊更糟）。📏 對重複的驗證嘗試設上限並放慢速度。

### 19. 信任請求主體中的身分
❌ `UpdateProfile(userId from body)`。✅ 從已驗證的權杖取得身分（本應用程式使用 `User.FindFirstValue`）。
🎯 攻擊者填入別人的 `userId` 就能編輯對方的帳號。📏 身分來自權杖，絕不來自酬載（payload）。

### 20. 把 JWT 存在 `localStorage`
❌ `localStorage.setItem('jwt', token)`。✅ 至少用 `sessionStorage`；最理想的是 httpOnly+Secure+SameSite 的 cookie + CSP。
🎯 任何 XSS 都能讀取 `localStorage` 並外洩權杖；它還會在關閉分頁後留存。📏 在可行的情況下，讓權杖遠離 JS 可讀取的儲存空間。

### 21. 伺服器端沒有密碼原則
❌ 只有 Angular 表單檢查強度。✅ 在伺服器端強制執行（本應用程式有 `PasswordPolicy.IsCompliant`）。
🎯 攻擊者直接呼叫 API，繞過用戶端驗證，把密碼設成「123」。📏 永遠以伺服器為準。

### 22. 冗長的驗證錯誤洩漏內部細節
❌ 回傳「DB timeout connecting to sql-prod-02」。✅ 通用訊息；細節只寫到伺服器記錄。
🎯 錯誤會洩漏基礎架構的拓撲，並協助攻擊者鎖定目標。📏 用戶端拿到通用訊息；記錄裡才有細節。

### 23. 工作階段固定／登入時未輪替權杖
❌ 登入後沿用驗證前的工作階段 id。✅ 在驗證成功時發放全新的權杖/工作階段。
🎯 攻擊者植入一個已知的工作階段 id，受害者登入後，攻擊者便可搭順風車。📏 權限變更時要換發新憑證。

### 24. 重設權杖：無期限／可重複使用／非一次性
❌ 重設連結永久有效，能用兩次。✅ 短 TTL、一次性、使用後即失效。
🎯 信箱裡的舊郵件/連結在數個月後被重放利用。📏 重設權杖必須短效且一次性。

## C. 授權與存取控制

### 25. 身分驗證 ≠ 授權
❌「使用者已登入」→ 放行。✅ 檢查他們是否被允許操作 *這個* 資源/動作。
🎯 任何已登入的使用者都能觸及只檢查了權杖的管理員動作。📏 驗證的是權限，而不只是權杖是否存在。

### 26. IDOR — 未檢查擁有權
❌ `GET /orders/{id}` 會回傳任何訂單。✅ 依擁有者限縮範圍：`WHERE Id=@id AND UserId=@me`。
🎯 使用者遞增 `id` 就能讀取其他使用者的紀錄。📏 每一次物件存取都要限縮在呼叫者的權限範圍內。

### 27. 只在用戶端做授權
❌ 在 Angular 裡把管理員按鈕藏起來就了事。✅ 在 API 上強制角色檢查（`[Authorize(Roles="Admin")]`）。
🎯 攻擊者直接呼叫端點；被隱藏的按鈕只是裝飾。📏 UI 的管制是使用者體驗；由伺服器來強制執行。

### 28. 缺少預設拒絕
❌ 除非你記得加 `[Authorize]`，否則端點都是公開的。✅ 全域後備原則要求驗證（本應用程式的 `FallbackPolicy`）。
🎯 新加的控制器因為有人忘了加屬性而以未受保護的狀態上線。📏 預設即安全；要退出保護必須 *明確* 為之。

### 29. 控制器層級的 `[AllowAnonymous]` 外溢到新動作
❌ 對整個控制器開放匿名。✅ 只限縮到單一動作（本應用程式在 `CoursePdfController.GetPdf` 上就是這麼做）。
🎯 之後加進控制器的動作會默默繼承匿名存取權。📏 匿名授權要逐一動作、狹窄地給予。

### 30. 信任來自用戶端的角色/聲明
❌ 從請求標頭/主體讀取 `isAdmin`。✅ 角色來自已簽章的權杖／伺服器儲存。
🎯 攻擊者設定 `X-Is-Admin: true`。📏 授權事實必須由伺服器驗證，而非由呼叫方自行宣稱。

### 31. 大量指派／過度張貼（over-posting）
❌ 把請求直接繫結到實體上，連 `IsAdmin`、`RoleIds` 都一起。✅ 繫結一個範圍狹窄的 DTO；以白名單列出欄位。
🎯 使用者張貼表單從未顯示的額外欄位（`isActive`、角色）而提升權限。📏 只接受該動作所擁有的欄位。

### 32. 水平與垂直權限的界線模糊
❌「編輯自己」與「編輯任何人」共用同一個處理常式。✅ 分開路徑；管理員路徑加上角色管制。
🎯 自助編輯端點被重複利用，只要改個 id 就能編輯別人。📏 把自助操作與管理員操作分開。

### 33. 忘了對次要動作做授權
❌ 保護了 `GET` 清單，卻沒保護一旁的 `export`/`bulk-delete`。✅ 對資源的每個動詞/動作都做授權。
🎯 被忽略的批次端點成了入侵的破口。📏 稽核 *所有* 動作，而不只是明顯的讀取。

## D. 注入攻擊

### 34. 用字串串接組成 SQL
❌ `"...WHERE Name='" + name + "'"`。✅ 使用參數（`@Name`）— Dapper 在本應用程式中全程如此。
🎯 `name = "' OR 1=1;--"` 會傾印/刪除整張資料表。📏 絕不用串接輸入的方式組 SQL。

### 35. 從輸入內插識別碼（資料表/欄位/排序）
❌ `$"ORDER BY {sortColumn}"`。✅ 以白名單限定為一組固定允許的欄位/排序方向。
🎯 參數無法繫結識別碼，於是開發者改用內插 → 透過 `sortColumn` 注入。📏 以允許清單驗證識別碼。

### 36. 未跳脫的 `LIKE` 萬用字元
❌ `LIKE '%'+@kw+'%'` 帶有未處理的 `%`/`_`。✅ 跳脫萬用字元 + `ESCAPE '\'`（本應用程式的 `SqlLike.EscapeWildcards`）。
🎯 搜尋 `%` 會比對到所有東西；精心構造的模式會造成病態掃描。📏 跳脫 LIKE 的中繼字元。

### 37. 作業系統命令注入
❌ `Process.Start("cmd", "/c convert " + userFile)`。✅ 以陣列傳遞引數；驗證/白名單化輸入。
🎯 `userFile = "a & del /q C:\\*"` 會執行任意命令。📏 絕不把不可信的輸入傳給 shell 字串。

### 38. 檔案存取的路徑穿越
❌ `File.ReadAllBytes(base + userPath)`。✅ 正規化 + 驗證解析後的路徑仍位於根目錄之下。
🎯 `userPath = "..\\..\\appsettings.json"` 會讀到機密。📏 先解析，再限制在允許的目錄內。

### 39. 透過原始 HTML 接收點造成 XSS
❌ `[innerHTML]="userContent"` / `dangerouslySetInnerHTML`。✅ 使用內插（Angular 預設會跳脫）。
🎯 被儲存的 `<img onerror=...>` 會在每位檢視者的工作階段中執行。📏 別把不可信的字串交給 HTML 接收點。

### 40. 繞過框架的 XSS 防護
❌ 對不可信的輸入使用 `DomSanitizer.bypassSecurityTrustHtml(x)`。✅ 只對你自己產生/信任的內容才繞過。
🎯 那一個「相信我」的呼叫，就是在原本安全的應用程式中開的 XSS 漏洞。📏 把每一次繞過都視為一項安全審查項目。

### 41. 開放式重新導向
❌ 直接用查詢參數 `return Redirect(returnUrl)`。✅ 只允許相對路徑／一份主機允許清單。
🎯 `returnUrl=https://evil.tld` 會借你的網域助長極具說服力的釣魚攻擊。📏 驗證重新導向的目標。

### 42. 伺服器端範本／運算式注入
❌ 算繪由使用者輸入拼成的範本字串。✅ 把資料以參數傳入，而非塞進範本原始碼。
🎯 輸入變成範本程式碼並在伺服器上執行。📏 資料就是資料；絕不把使用者輸入編譯成範本。

### 43. LDAP / NoSQL / 標頭注入
❌ 把輸入串接進 LDAP 篩選器、Mongo 查詢或 HTTP 標頭。✅ 依目標語法進行編碼/參數化。
🎯 `*)(uid=*` 可繞過 LDAP 驗證篩選器；標頭中的 CRLF 會拆分回應。📏 永遠針對特定的接收點做跳脫。

### 44. 對輸入使用 `eval`/`Function`/動態程式碼
❌ `eval(userExpr)`、`new Function(userStr)`。✅ 為允許的語法寫一個剖析器/白名單。
🎯 任意程式碼會在伺服器或受害者的瀏覽器中執行。📏 絕不執行源自輸入的字串。

### 45. XML 外部實體（XXE）
❌ 對不可信的 XML 使用預設、會處理 DTD 的 `XmlReader`/剖析器。✅ 停用 DTD/外部實體。
🎯 `<!ENTITY xxe SYSTEM "file:///...">` 會讀取本機檔案或觸發 SSRF。📏 在剖析器上關閉外部實體。

### 46. 匯出時的 CSV／公式注入
❌ 把以 `=`、`+`、`-`、`@` 開頭的使用者儲存格直接寫進 CSV。✅ 在前面加上 `'` 或清理開頭的運算子。
🎯 Excel 一開啟就執行那個「公式」→ 在受害者的機器上外洩資料。📏 在匯出中消除公式的觸發字元。

## E. 輸入驗證與反序列化

### 47. 只信任用戶端驗證
❌ Angular 檢查過了，伺服器就假設它是乾淨的。✅ 在伺服器端重新驗證每個欄位。
🎯 直接呼叫 API 會完全跳過表單。📏 用戶端驗證是使用者體驗；伺服器驗證才是真相。

### 48. 未設上限的輸入大小
❌ 接受任意長度的字串/陣列/檔案。✅ 對長度、數量以及請求/主體大小設上限。
🎯 一個 500 MB 的「name」或 10⁶ 個元素的陣列會耗盡記憶體／讓 PDF 暴增。📏 每個輸入都有明確的上限。

### 49. 不安全的反序列化
❌ 對不可信的資料使用 `BinaryFormatter`／內嵌型別的反序列化器。✅ 使用 `System.Text.Json` 搭配已知型別。
🎯 精心構造的酬載會實例化 gadget 型別 → 遠端程式碼執行（RCE）。📏 反序列化到固定的 DTO；絕不讓輸入決定多型型別。

### 50. 讓輸入控制 `JsonSerializer` 的 `TypeNameHandling`／多型
❌ 讓酬載挑選 CLR 型別。✅ 繫結到具體的 DTO；以允許清單驗證鑑別子（discriminator）。
🎯 攻擊者指名一個危險的型別來建構。📏 由伺服器選擇型別，而不是 JSON。

### 51. 剖析後未做 null／再次檢查
❌ 假設 `dict[key]`/`FindFirst(...)` 一定非 null。✅ 檢查 null 並採失敗即拒絕。
🎯 遺漏的聲明/欄位會擲出例外，或更糟——被當成「空的但有效」的值。📏 使用前先驗證其存在。

### 52. 數值溢位／未檢查的轉型
❌ `(int)longValue`，在 `unchecked` 中悄悄回繞。✅ 驗證範圍；在關鍵處使用 `checked`。
🎯 一個超大的數量回繞成負數，因而跳過了上限檢查。📏 對來自輸入的數字設界並做範圍檢查。

### 53. 對不可信輸入使用會災難性回溯的正規表示式（ReDoS）
❌ 對使用者字串套用 `(a+)+$` 這類模式。✅ 線性模式；逾時；在可用時使用錨定、佔有式（possessive）量詞。
🎯 一個精心構造的字串就能讓一個 CPU 核心卡上數秒/數分鐘。📏 讓面向輸入的正規表示式保持線性 + 有時間上限。

### 54. 信任 Content-type／檔案類型
❌ 信任上傳的 `Content-Type` 或副檔名。✅ 嗅探魔術位元組（magic bytes）；存放在 webroot 之外；重新編碼圖片。
🎯 一個實際上是可執行檔/HTML 的 `.jpg` 被提供並執行。📏 依內容而非標籤來驗證檔案類型。

## F. 錯誤處理與記錄

### 55. 把堆疊追蹤／SQL 錯誤洩漏給用戶端
❌ 回傳例外訊息。✅ 給用戶端通用訊息，把完整細節寫到伺服器記錄（本應用程式的 `ExceptionHandlingMiddleware`）。
🎯 錯誤會透露資料表名稱、路徑、版本——偵察的金礦。📏 詳細的記錄、通用的回應。

### 56. 吞掉例外
❌ `catch { }`。✅ 處理或重新擲出；連同上下文一起記錄。
🎯 一項失敗的安全檢查被默默忽略，請求照樣進行。📏 絕不空 catch；失敗必須被看見。

### 57. 過度廣泛地捕捉 `Exception` 並繼續執行
❌ 全面捕捉，把 `OperationCanceledException`、驗證失敗都掩蓋掉。✅ 捕捉特定型別；讓致命的例外浮現。
🎯 一個取消或授權例外被當成可復原的小狀況。📏 狹窄地捕捉；別藏錯了東西。

### 58. 記錄未清理的輸入（記錄偽造）
❌ `LogInformation(userInput)` 帶有原始換行字元。✅ 記錄結構化欄位；由接收端負責跳脫。
🎯 輸入中的 CRLF 會注入假的記錄行以掩蓋行蹤。📏 使用結構化記錄，而非字串串接的訊息。

### 59. 敏感異動沒有稽核軌跡
❌ 對特權資料做無聲的新增/更新/刪除。✅ 記錄何人/何事/何時（本應用程式的 `RowAuditWriter`）。
🎯 入侵事件無法還原；內部人員的行為可以矢口否認。📏 稽核特權狀態的變更。

### 60. 錯誤驅動的旁通道
❌ 對「存在」與「禁止」給出不同的回應/回應時間。✅ 兩者都給一致的 404/回應。
🎯 回應的差異會洩漏隱藏資源的存在。📏 別讓錯誤揭露你正在隱藏的東西。

### 61. 記錄完整的請求/回應主體
❌ 在 Info 層級傾印每個酬載。✅ 記錄中繼資料；遮蔽主體，尤其是驗證/PII。
🎯 密碼/權杖/PII 在記錄裡越積越多，並隨記錄一起外洩。📏 主體是敏感的；別整批記錄下來。

### 62. 在完整性/授權檢查失敗後仍繼續執行
❌ 記錄「檢查失敗」後照樣繼續。✅ 失敗即拒絕——停下並回傳錯誤。
🎯 那項檢查淪為裝飾；危險的路徑照樣執行。📏 失敗的安全檢查必須阻止該操作。

## G. .NET / C# 陷阱

### 63. `async void`
❌ `async void DoWork()`。✅ `async Task`（事件處理常式除外）。
🎯 例外無法被呼叫者捕捉並會讓行程崩潰；也沒有東西能 await 它。📏 用 `async Task`，絕不用 `async void`。

### 64. 在熱路徑上使用 `.Result` / `.Wait()` / `.GetAwaiter().GetResult()`
❌ 在請求處理中對非同步進行阻塞。✅ 一路向上 `await`。
🎯 執行緒池耗竭 → 在負載下產生死結與停頓。📏 在伺服器程式碼中別用同步包非同步（sync-over-async）。

### 65. 未傳遞 `CancellationToken`
❌ 在非同步的 DB/HTTP 呼叫中忽略權杖。✅ 一路貫穿傳遞（本應用程式把它傳給每個 `CommandDefinition`）。
🎯 被放棄的請求持續執行，浪費資料庫連線與 CPU。📏 傳播呼叫者的取消權杖。

### 66. 每個請求都建立 `HttpClient`（socket 耗竭）
❌ 每次呼叫都 `new HttpClient()`。✅ `IHttpClientFactory`／共用的用戶端。
🎯 Socket 卡在 `TIME_WAIT`；主機在負載下用光連接埠。📏 透過工廠重複使用 `HttpClient`。

### 67. 從單例（singleton）持有/釋放具範圍（scoped）的資源
❌ 在單例中擷取具範圍的 `DbConnection`/DbContext。✅ 依每次操作解析；每次使用時開啟/關閉。
🎯 跨請求的狀態滲漏、執行緒錯誤、連線洩漏。📏 尊重 DI 的生命週期；別擷取生命週期較短的服務。

### 68. 未釋放 `IDisposable`
❌ 忘了釋放連線/串流。✅ `using var ...`（本應用程式在連線上就是這麼用）。
🎯 洩漏的連線耗盡連線池；應用程式為了等一條連線而卡住。📏 對每個可釋放物件都用 `using`。

### 69. 可變的 `static` 共用狀態
❌ 一個每次請求都會被變更的 `static` 字典/清單。✅ 每次請求的範圍，或使用執行緒安全/不可變的結構。
🎯 競爭條件會損毀資料，或把一位使用者的資料洩漏給另一位。📏 伺服器中不得有未同步的可變靜態變數。

### 70. 用 `DateTime.Now` 而非 `DateTime.UtcNow`
❌ 儲存/比較本地時間。✅ 全程使用 UTC；只在邊界處為了顯示才轉換。
🎯 日光節約時間/時區偏移會破壞權杖到期、排程與比較。📏 以 UTC 儲存與比較。

### 71. 對文化敏感的剖析/格式化/比較
❌ 使用當前環境文化的 `double.Parse(s)` / `ToUpper()`。✅ 邏輯上使用 `InvariantCulture`；`StringComparison.Ordinal`。
🎯 以逗號作小數點的地區設定會剖析錯誤；土耳其文的 I 會破壞識別碼比較。📏 機器邏輯一律用不變（invariant）/序數。

### 72. 用浮點數處理金額
❌ `double price`。✅ `decimal`。
🎯 捨入誤差讓總額差了幾分錢，日積月累導致對帳失敗。📏 金額用 `decimal`。

### 73. 具現化後才 `.Select().Where()`（N+1／用戶端求值）
❌ 載入所有列後在記憶體中篩選／在迴圈中逐項查詢。✅ 在 SQL 中篩選；批次處理。
🎯 在真實資料下會有數千次來回或全表載入。📏 把篩選/join 下推到資料庫。

### 74. 在函式庫中忽略 `ConfigureAwait`／不必要地擷取上下文
❌ 假設存在同步上下文的函式庫程式碼。✅ 在函式庫層使用 `ConfigureAwait(false)`。
🎯 被綁定上下文的呼叫者使用時會產生死結。📏 函式庫的 await 不擷取上下文。

## H. Dapper / SQL Server / 資料層

### 75. 多陳述式寫入卻沒有交易
❌ 兩次 `ExecuteAsync` 呼叫，沒有交易。✅ 包在單一交易中（本應用程式在新增/更新/刪除時就是這麼做）。
🎯 陳述式之間的當機會留下寫到一半、不一致的資料。📏 相關的寫入要具原子性。

### 76. 沒有樂觀並行控制
❌ 盲目的 `UPDATE ... WHERE Id=@id`。✅ 檢查 `rowversion`/時間戳記；在 `affected == 0` 時回傳 409。
🎯 兩位管理員編輯同一列；第二位默默覆寫了第一位（更新遺失）。📏 用版本來保護更新。

### 77. 先讀後寫卻沒有鎖（競爭）
❌ 在有競用的情況下 `SELECT slot; compute; UPDATE`。✅ `UPDLOCK, HOLDLOCK` / `SERIALIZABLE` / 唯一性條件約束。
🎯 兩個並行請求都通過檢查並都寫入，違反了不變量。📏 對共用列的「先檢查再動作」要序列化。

### 78. 以為資料庫的唯一/外鍵錯誤絕不會發生
❌ 沒有處理 2601/2627/547。✅ 對應成 409/400 並附上乾淨的訊息。
🎯 一個重複鍵以原始的 500 浮現，洩漏 SQL 細節。📏 把已知的 SQL 錯誤轉譯成 HTTP 語意。

### 79. `SELECT *`／投影敏感欄位
❌ `SELECT *` 會把 `PasswordHash` 一起撈出來。✅ 明確列出欄位；絕不選取機密（本應用程式會省略 `PasswordHash`）。
🎯 雜湊/機密會搭順風車進到 DTO/記錄/稽核中。📏 只投影你打算公開的欄位。

### 80. 未設上限的清單端點（沒有分頁）
❌ 回傳每一列。✅ `OFFSET/FETCH` 或 `TOP (@max)` + 總筆數。
🎯 一張長大的資料表回傳數百萬列，讓 API 記憶體不足（OOM）並使 UI 停頓。📏 每個清單都分頁；並設上限。

### 81. 對遠端資料庫使用 `TrustServerCertificate=True` / `Encrypt=False`
❌ 把開發用的連線設定帶到正式環境。✅ 對非本機資料庫加密 + 驗證憑證。
🎯 路徑上的攻擊者可讀取/竄改資料庫流量（僅同主機的 SQLEXPRESS 才安全）。📏 跨主機時要加密 + 驗證憑證。

### 82. 權限過高的資料庫帳號
❌ 應用程式以 `sa`/db_owner 連線。✅ 最小權限登入（只對所需資料表做 CRUD）。
🎯 一次注入或臭蟲就從局部損害升級為整台伺服器被攻陷。📏 應用程式的資料庫使用者只有最低限度的權限。

## I. Angular / 前端 / TypeScript

### 83. 機密出現在前端套件包中
❌ 把 API 金鑰/機密放在 `environment.ts`。✅ 機密留在伺服器端；SPA 只持有公開的設定。
🎯 任何人打開開發者工具／讀取 JS 套件包就能取走金鑰。📏 任何機密都不送到瀏覽器。

### 84. 把 bearer 權杖附加到跨來源請求上
❌ 攔截器對每個 URL 都加上 `Authorization`。✅ 只對同來源／允許清單內的 API（本應用程式有做範圍限制）。
🎯 一個發往第三方 URL 的請求會把你的權杖洩漏給對方。📏 只把憑證送往你自己的 API。

### 85. `any` 抹除型別安全
❌ 讓 `data: any` 貫穿整個應用程式。✅ 使用真正的介面/DTO。
🎯 後端結構的變更編譯時毫無問題，卻在執行期當著使用者的面爆掉。📏 為你的 API 合約標註型別。

### 86. 非 null 斷言（`!`）掩蓋了真正的 null
❌ `this.user!.name`。✅ 使用防衛、選擇性串連（optional chaining），或好好處理 null。
🎯 一個真的不存在的值會在正式環境擲出 `undefined is not an object`。📏 `!` 是一個你必須真正兌現的承諾。

### 87. 永不取消的訂閱
❌ 在元件中 `.subscribe(...)` 卻沒有拆解（teardown）。✅ `takeUntilDestroyed` / async pipe / `DestroyRef`。
🎯 記憶體洩漏 + 導覽後重複的處理常式仍在觸發。📏 每個手動訂閱都需要一個拆解機制。

### 88. 商業規則只在 UI 上強制執行
❌ 把「停用」按鈕當成唯一的防線。✅ API 也要拒絕該無效操作。
🎯 一個精心構造的請求會無視停用狀態。📏 UI 什麼都無法強制執行；必須由伺服器來做。

### 89. 把伺服器資料當成安全的 HTML 來信任
❌ 假設後端字串是乾淨的，就透過 `[innerHTML]` 算繪它們。✅ 預設跳脫；若確實需要 HTML 就先淨化。
🎯 儲存型 XSS 來自 *較早* 的端點未經淨化就接受的資料。📏 在接收點把所有資料都視為不可信。

### 90. 用字串串接組出 URL（前端）
❌ `` `/api/x/${id}` `` 帶有未編碼的 `id`。✅ `encodeURIComponent` / `HttpParams`（本應用程式會對路由 id 編碼）。
🎯 特殊字元會破壞路由或使參數走私（parameter smuggling）成為可能。📏 對每個動態 URL 區段編碼。

### 91. 透過冗長的用戶端記錄洩漏資料
❌ 在正式環境留下含權杖/PII 的 `console.log(response)`。✅ 從正式版建置中移除除錯記錄。
🎯 敏感資料留在瀏覽器主控台與錯誤追蹤器中。📏 正式環境中不得有含敏感資料的 `console.*`。

### 92. 儲存可在用戶端偽造的衍生驗證狀態
❌ 把 `isAdmin` 旗標放在使用者可以編輯的儲存區。✅ 從權杖推導 UI；伺服器對每次呼叫重新檢查。
🎯 使用者在記憶體中翻轉旗標便解鎖管理員 UI（伺服器端仍會擋下——但別只靠這一點）。📏 用戶端的驗證狀態是提示，不是關卡。

## J. 設定、部署、基礎設施與供應鏈

### 93. 正式環境啟用除錯/冗長錯誤
❌ 在正式環境開放開發者例外頁面／Swagger。✅ 僅限開發環境（本應用程式以 `IsDevelopment()` 管控 Swagger）。
🎯 完整的 API 介面 + 堆疊追蹤被交到匿名呼叫者手上。📏 除錯工具只留在開發環境。

### 94. 萬用字元 CORS 搭配憑證
❌ `AllowAnyOrigin()` + `AllowCredentials()`。✅ 明確的來源允許清單（本應用程式限制在 loopback）。
🎯 任何網站都能以你的使用者身分發出已驗證的呼叫。📏 絕不把萬用字元來源與憑證搭配使用。

### 95. 正式環境沒有 TLS / HSTS
❌ 以 HTTP 提供 API。✅ 在非開發環境使用 `UseHttpsRedirection` + `UseHsts`（本應用程式有做）。
🎯 權杖/密碼以明文傳輸；路徑上的攻擊者可讀取它們。📏 除了本機開發外，處處強制 HTTPS。

### 96. 缺少安全標頭 / CSP
❌ 沒有 `Content-Security-Policy`、`X-Content-Type-Options` 等。✅ 在邊界/應用程式端加上它們。
🎯 XSS 酬載暢行無阻；MIME 嗅探會把一個上傳檔變成指令碼。📏 交付一組基準的安全標頭。

### 97. 未釘選／不可信的 CI 動作與相依套件
❌ `uses: some/action@main`、浮動的相依版本範圍。✅ 以 SHA 釘選動作；釘選 + 稽核相依套件；提交鎖定檔（lockfile）。
🎯 一個被入侵的上游標籤會帶著你的機密在你的流水線中執行。📏 釘選會執行的東西；稽核你拉取的東西。

### 98. 機密以明文環境變數出現在 CI 記錄／映像檔中
❌ `echo $SECRET`、把機密烤進 Docker 層。✅ 遮罩式的機密儲存；執行期注入；映像檔中不放機密。
🎯 機密在建置記錄或 `docker history` 中外洩。📏 機密在執行期注入，絕不記錄或包進映像層。

### 99. 容器以 root 執行／使用 `latest` 基礎映像
❌ 沒有 `USER`、`FROM node:latest`。✅ 非 root 的 `USER`；釘選、已修補的基礎映像摘要（digest）。
🎯 容器逃逸會以 root 落地；`latest` 會默默拉取有漏洞/被植入後門的映像。📏 最小權限 + 釘選的基礎映像。

### 100. 含真實憑證的 `.env`／設定被納入 git 追蹤
